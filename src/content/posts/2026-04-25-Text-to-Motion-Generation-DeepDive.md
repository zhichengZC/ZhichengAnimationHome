**HumanML3D 263 维运动向量**（motion 表示）：
- 根节点速度 4 + 关节位置 63 + 关节速度 63 + 关节 6D 旋转 126 + 脚接触 4 + 其他 3 = 263
- 不理解这 263 维没关系，**你只需要把 motion 序列想成一个形状为 `(T, 263)` 的数字矩阵**，T 是帧数（典型 T=196，即 ~8 秒 @ 24fps）
- 来源：Guo et al., CVPR 2022 HumanML3D 数据集

## §1 三条路的地图 [L0 核心]

### 1.1 开场场景：一个周一上午的需求

周一上午，策划敲你工位隔板：

> "我想要角色做'走得很累的样子'的动作，能不能用 AI 生一段我看看？别等动捕了。"

你打开电脑，想到脑子里现有的 ML 工具有 GPT、Diffusion、BERT 这些大家族。**它们能不能直接用来生成动作？**

答案是能，但**有三条完全不同的路**，各有各的起点和坑。这三条路分别是：

| 路线 | 代表方法 | 核心思路 |
|---|---|---|
| 🅰️ **自回归路线** | T2M-GPT (CVPR 2023) | 把动作编成"外语词"，用 GPT 一个词一个词"说"出来 |
| 🅱️ **扩散路线** | MDM (ICLR 2023) | 在连续向量空间从噪声逐步"去噪"出动作 |
| 🅲️ **并行 masked 路线** | MoMask (CVPR 2024) | 用类似 BERT 的方式一次填空多个位置，迭代几步就完 |

三条路面对同一个输入（文字）、同一个输出（motion 序列），但**走法完全不同**。走法不同意味着：训练成本、推理速度、生成质量、能做什么额外功能（编辑？inbetween？）全都不同。

### 1.2 三条路在"一次推理"里做了多少事

让我们直接看**最硬的指标**：生成一段 T=196 帧的 motion，三条路分别要跑多少次网络前向？

**🅰️ T2M-GPT**：自回归，每次预测下一个 token
- 196 个 token → 需要 **196 次前向**
- 每次前向都带有 self-attention，时间复杂度 O(T²)
- 实测 ~1-2 秒（V100 GPU）

**🅱️ MDM**：扩散，每一步都对整段 motion 去噪
- 标准 DDPM T=1000 步
- 每步都是对完整 (T=196, D=263) 的张量做一次 Transformer 前向
- 实测 ~30 秒（V100 GPU）

**🅲️ MoMask**：masked 并行，一次填多个 token
- Base transformer 迭代 10 步 + Residual transformer 迭代 5 步 = **15 步**
- 每步同时填多个位置，而非一个一个
- 实测 ~0.15 秒（V100 GPU）

### 1.3 Hand-calc 1 —— 三条路的推理时间直觉 🔢

**题目**：假设每次 Transformer 前向固定 10ms（简化，实际与 T 相关），生成 T=196 帧 motion 需要多少时间？

| 路线 | 前向次数 | 估算时间 | 实测数据 |
|---|---|---|---|
| T2M-GPT 自回归 | 196 次 | 1.96 秒 | 1-2 秒 |
| MDM 扩散 | 1000 次 | 10 秒 | 30 秒（实际每步 > 10ms） |
| MoMask 并行 | 15 次 | 0.15 秒 | 0.15 秒 |

**直觉结论**：
- MoMask 比 MDM **快约 67 倍**
- MoMask 比 T2M-GPT **快约 13 倍**
- **时间差异的核心是"前向次数"**，不是模型大小

<details>
<summary>自测：如果 T=400 帧（约 16 秒），三条路时间比例会变化吗？</summary>

T2M-GPT 线性增长（400ms），MDM 保持 1000 步但每步计算量因 T 变大（Transformer 注意力是 O(T²)），MoMask 同样保持 15 步。**MoMask 随 T 增长速度最慢**，所以长动作里 MoMask 优势更明显。

</details>

*图 1：三条路在同一个任务上的地图。相同的文字输入、相同的 motion 输出，中间走了完全不同的路径。*

![三条路推理次数对比图](/images/Text-to-Motion-Generation-DeepDive/fig2.png)

*图 2：三条路生成 T=196 帧 motion 所需前向次数直观对比。MDM 的 1000 步是"不可接受"的工程代价，MoMask 的 15 步是工业产品级。*

### 1.4 为什么同一个任务会有这么多路？

这不是工程师"有钱任性"，而是三条路各自发现了 motion 数据的不同属性：

- **T2M-GPT** 发现：motion 经过 VQ-VAE 编码后，其 token 序列和自然语言惊人相似，可以直接套 GPT。**代价**：离散化本身有损，且自回归串行慢。
- **MDM** 发现：motion 是连续向量，天生适合扩散模型（扩散是为连续信号设计的）。**代价**：为了生成质量只能走 1000 步，慢到不能工业化。
- **MoMask** 发现：Motion 离散化 + masked modeling 可以同时拿到"离散的紧凑性"和"并行的速度"。**代价**：训练 pipeline 复杂（要训 RVQ + base transformer + residual transformer 三样）。

**接下来三章我们各走一遍，看每条路的具体细节。**

### 1.5 本章自测

<details>
<summary>MoMask 比 MDM 快 67 倍，但它的模型一定比 MDM 小吗？</summary>

不一定。MoMask 可能有更多参数（RVQ + base + residual），但**推理步数**少得多。**速度和参数量不是同一件事**，推理步数才是瓶颈。

</details>

<details>
<summary>如果你要实时交互（比如 NPC 即时响应），三条路哪个完全不可行？</summary>

MDM 的 30 秒推理完全不可行。T2M-GPT 的 1-2 秒接近边缘（可以做"对话间隙触发"）。MoMask 的 0.15 秒是唯一真正实时的选择。

</details>

## §2 第一条路：T2M-GPT — 把动作当外语 [L0 核心] ⭐

### 2.1 开场问题：动作真的是一种"语言"吗？

如果给你一段英语文本 `"the quick brown fox jumps over"`，GPT 可以预测下一个词是 `"the"`。这整套 "prefix → next token" 的自回归机制成熟到工业级。

**问题是**：动作不是词，动作是连续的 263 维向量序列，GPT 没法直接吃。

**T2M-GPT 的思路**：
> "如果能把连续的 motion 先**离散化**成 token（就像把音频离散化成音符），那我们就可以用成熟的 GPT 框架直接生成。"

这就是 **2 阶段架构**的由来：
1. **Stage 1** — 用 VQ-VAE 把 motion 编码成离散 token（"学动作的词典"）
2. **Stage 2** — 用 GPT-2 学这些 token 的语法（"学怎么把词连成句"）

![T2M-GPT两阶段架构图](/images/Text-to-Motion-Generation-DeepDive/fig3.png)

*图 3：T2M-GPT 两阶段。第一阶段用 VQ-VAE 把 motion 压成 token 序列（相当于"动作词典"），第二阶段用 GPT-2 学这些 token 在文本条件下的自回归分布。*

### 2.2 Stage 1：VQ-VAE 怎么把 motion 变成 token（快速回顾）

昨天 VQ-VAE Principles 已经讲透，这里只要记住三件事：

1. **Encoder** 把 motion `(T=196, D=263)` 下采样成 `(T/4=49, D=512)` 的特征
2. **Codebook** K=512 个学到的向量，每个 z_e 找**最近邻** codebook entry，变成 one-hot token id
3. **Decoder** 反过来把 token 解码回 motion

**关键数字**：
- codebook size K=512（类比：英语词汇表 ~50000；中文 ~5000；**motion 词汇只要 512 就够**）
- 下采样比 4（8 秒动作 = 196 帧 → 49 个 token，像一句话 49 个词）

**为什么 K=512？**
- Motion 的"词汇"是低级运动原语（左脚迈一步 / 右脚抬起 / 手臂摆动）
- 高层语义（走、跑、跳）由这些原语**组合**出来
- K=512 是 T2M-GPT 的设计选择和折中——对于覆盖 HumanML3D 的基本运动类型基本够用，但 VQ-VAE 在 K=512 时的重建误差比较显著（这也是 MoMask 用 RVQ 6 层 × 512 = 3072 个等效码字来弥补的原因，详见 §4.2 和 `2026-04-24-VQVAE-Principles` §7.2）

### 2.3 Stage 2：GPT-2 怎么学 motion token 的"语法"

**输入序列**：`[文本 context] + [motion token 1] + [motion token 2] + ... + [motion token 49]`

**训练目标**：给定前 i 个 token + 文本，预测第 i+1 个 token。（和英语 GPT 完全相同的 next-token prediction）

**文本 context 怎么进来？**
- T2M-GPT 用 CLIP-ViT-B/32 的 text tower 把文字变成 512 维向量
- 把这个向量**prepend 到 motion token 序列前面**，让 GPT 的 self-attention 同时看到文字和已生成的 motion token
- 可以理解为：给 GPT 一个"提示词"，让它顺着这个提示词说动作

![T2M-GPT自回归单步示意图](/images/Text-to-Motion-Generation-DeepDive/fig4.png)

*图 4：T2M-GPT 的自回归单步。给定 text context 和已生成的前 i 个 motion token，GPT-2 预测第 i+1 个 token 的概率分布，从中采样得到下一个 token，再接回输入。像打字一样一个一个吐。*

### 2.4 Hand-calc 2 —— 自回归一次完整推理的代价 🔢

**题目**：T=196 帧 motion → 49 个 token。GPT-2 生成一个完整 motion 需要多少次 self-attention？

**每次前向的注意力复杂度**：O(L²)，L 是当前序列长度

**总代价**：
- 生成第 1 个 token：注意力 O(1²) = 1
- 生成第 2 个 token：注意力 O(2²) = 4
- ...
- 生成第 49 个 token：注意力 O(49²) = 2401

**累计** ≈ Σ i² (i=1 to 49) ≈ 40000 单位的注意力计算

对比：非自回归一次性处理 49 个 token 只要 1 次 O(49²) = 2401 单位。**自回归因为要 49 次前向，代价是一次性的 ~17 倍**。

**直觉结论**：**自回归的天然成本** = 序列越长、代价越是平方级增长。这就是后面 MoMask 要解决的痛点。

### 2.5 T2M-GPT 的优缺点

**优点**：
- 训练稳定（GPT-2 框架成熟）
- 模型小（~100M 参数）
- 可以用 Top-k / Temperature / Nucleus sampling 直接控制多样性

**缺点**：
- **推理慢**（自回归串行）
- **codebook 崩塌风险**（昨天 VQ-VAE Principles §5 讲过）
- **质量上限受 VQ-VAE 编码损失限制**（VQ 重建误差 = 生成上限）

### 2.6 章末自测

<details>
<summary>为什么 T2M-GPT 的 codebook 只用 512 而英语 GPT 要 50000？</summary>

**语言粒度不同**。英语 GPT 需要区分每个单词（单词间语义差距大），motion 只需要区分**运动原语**（几百种足够）。这也是为什么 motion 数据集更小也能训好。

</details>

<details>
<summary>如果 VQ-VAE 的 codebook 崩塌了，T2M-GPT 的生成会怎么表现？</summary>

会出现"**动作重复**"或"**动作僵硬**"——因为可用的动作词汇就那几个，GPT 再怎么组合也走不出这个小集合。**T2M-GPT 的生成质量天花板 = VQ-VAE 的重建质量**。

</details>

## §3 第二条路：MDM — 把 Diffusion 搬到动作 [L0 核心] ⭐

### 3.1 开场问题：Motion 是连续的，直接扩散行不行？

上一节 T2M-GPT 绕了一圈："motion → token → GPT → token → motion"，每一步都损失一些信息。

**有没有更直接的路？**

Motion 本来就是连续的 `(T, 263)` 向量。而我们昨天学过，Diffusion 天生是为**连续信号**设计的（图像像素值 / 音频波形）。

**MDM 的思路**：
> "绕什么绕？motion 向量就是信号，直接套 DDPM 加噪 → 去噪。"

### 3.2 DDPM 三件事的 Motion 版（快速引用）

先回忆 4 月 23 日 Diffusion 的三件事（**不重推公式**）：

**前向加噪 q**：
$$x_0 \to x_1 \to x_2 \to \dots \to x_T$$
每一步加一点高斯噪声，T=1000 步后 `x_T` 变纯噪声

**反向去噪 p_θ**：
$$x_T \to x_{T-1} \to \dots \to x_0$$
神经网络 **ε_θ(x_t, t)** 预测 t 时刻的噪声，扣掉得到 x_{t-1}

**Loss**：MSE on predicted noise
$$L = \mathbb{E}_{t, x_0, \epsilon} \|\epsilon - \epsilon_\theta(x_t, t)\|^2$$

**Motion 版的唯一区别**：`x_t` 不是图像，而是形状 `(T=196, D=263)` 的 motion 张量。加噪去噪完全同套数学。

![MDM扩散过程示意图](/images/Text-to-Motion-Generation-DeepDive/fig5.png)

*图 5：MDM 把一段完整 motion 当作连续信号，每步加一点噪声（正向），或每步减一点噪声（反向）。和图像扩散唯一不同：张量形状是 `(T, 263)` 而非 `(H, W, 3)`。*

### 3.3 网络架构：带时间注意力的 Transformer 变体（非标准 Transformer Decoder）

图像扩散用 U-Net 是因为图像有空间局部性（附近像素相关）。motion 序列虽然时间上也有相关性，但 MDM 用的是一个修改过的 Transformer 架构（带 temporal attention 的变体，而非标准 Transformer Decoder）更自然 ：
- **输入**：`(T=196, D=263)` motion 张量 + 时间步 t（用 sinusoidal embedding）+ 文本 condition（CLIP 向量 prepend 到序列前面）
- **输出**：同形状 `(T=196, D=263)` 的预测噪声 ε

网络结构就是 Transformer 堆叠，8 层左右够。

### 3.4 Classifier-Free Guidance（CFG）：controlling strength

**问题**：训练好的 ε_θ(x_t, t, c) 给定文本 c 能输出"一段符合文本的动作"，但有时它生成的动作"和文本关联度不够强"。

**CFG 的办法**：训练时随机 10% 把 c 设为 ∅（无条件），让同一个网络既会"有条件生成"也会"无条件生成"。推理时组合两个预测：

$$\boxed{\epsilon_{\text{final}} = \epsilon_\theta(x_t, \varnothing) + w \cdot [\epsilon_\theta(x_t, c) - \epsilon_\theta(x_t, \varnothing)]}$$

**这个公式的直觉**：
- 无条件预测 = "一段自然动作应该长什么样"（默认分布）
- 有条件预测 = "一段符合 `c` 的动作应该长什么样"（偏离默认的方向）
- 差值 = "往文本 c 方向拉的力"
- w = 拉力强度

### 3.5 Hand-calc 3 —— CFG scale 的调参直觉 🔢

**题目**：设 w=1, 2.5, 7.5 三种情况，直观会发生什么？

| w 值 | ε_final 构成 | 效果 |
|---|---|---|
| w = 0 | 纯无条件 | 完全随机动作，文字忽略 |
| w = 1 | 只有条件预测 | 温和响应文字，多样性好 |
| w = 2.5 | 无条件 + 1.5× 差值 | 响应文字更强，多样性稍损（默认） |
| w = 7.5 | 无条件 + 6.5× 差值 | 极端响应文字，动作开始变 "过度夸张"、不自然 |

**实测**：
- MDM 原论文：w=2.5 是默认
- 用户调到 w=5-7.5 可以得到"更忠实于文本但动作较生硬"的版本
- 超过 10 几乎必崩（动作崩塌成几种 stereotypical 姿态）

![CFG在motion扩散中的方向调制示意图](/images/Text-to-Motion-Generation-DeepDive/fig6.png)

*图 6：CFG 在 motion 扩散中的方向调制。w 像是把"无条件生成的自然动作"拉向"有条件（文字）方向"的杠杆，调大贴文字但牺牲自然度。*

### 3.6 MDM 的优缺点

**优点**：
- **生成质量可扩展**（1000 步有足够空间细修）
- **天然支持编辑**（inbetween、局部 edit 直接就能做，因为扩散过程本身可逆）
- **不需要 VQ 前处理**（无损信号）

**缺点**：
- **推理巨慢**（1000 步 × 每步 Transformer 前向 ~30ms = 30 秒）
- 每次生成只能出一段完整动作（不能流式输出）
- 训练收敛慢（扩散训练的通病）

### 3.7 章末自测

<details>
<summary>能不能把 MDM 的 1000 步降到 50 步？</summary>

可以，用 DDIM sampler 或 DPM-Solver 这种加速采样器。**但质量会明显下降**（扩散的"步数 ≈ 质量"是硬约束）。另一条思路是训一个新模型 consistency model 直接学"一步去噪"，但那是另一篇论文。

</details>

<details>
<summary>MDM 能做"把这段走路中途改成跳起来"的局部编辑。T2M-GPT 能做吗？</summary>

T2M-GPT **天然不能**。因为它是纯自回归：给定 prefix 只能往后续，不能从中间改。MDM 因为扩散过程把中间状态"展开"在时间维度上，你可以**把想保留的部分 noise-free，只对想改的部分扩散**，这是 MDM 相比 GPT 路线的一个独家能力。

</details>

> 📎 **衔接下一文档**：本文只讲了 MDM 的基本原理。明天（`2026-04-26`）的 MDM/MLD 深讲文档会展开 MDM 的训练细节、DDIM 加速采样、MLD（Motion Latent Diffusion）如何在潜空间做扩散来提速，以及 MotionDiffuse 的条件化控制。如果你对"扩散怎么搬到动作上"还有疑问，那里有答案。

## §4 第三条路：MoMask — 用 BERT 的思路并行填空 [L0 核心] ⭐

### 4.1 开场问题：MDM 的 1000 步能不能压成 15 步？

MDM 1000 步慢到工业不可用。T2M-GPT 的 196 步也不够快。

**关键观察**：BERT 不是一个一个预测 token，而是**一次预测一批被 mask 的 token**。如果把这个思路搬到 motion：

**MoMask 的思路**：
> "先把 motion 离散成 token（像 T2M-GPT），但不要自回归。改用 BERT 式的 masked prediction，一次填一批，迭代 10 次就完。"

### 4.2 三大组件

**组件 1：RVQ 6 层码本**（回顾 2026-04-24 §7）
- 第 1 层：学 motion 最主干的信息（大动作方向）
- 第 2 层：学第 1 层的**残差**（细节）
- ...
- 第 6 层：学前 5 层的残差（最细的抖动）
- 总共 6 × 512 = 3072 个可用的"动作词"（比 T2M-GPT 的 512 丰富 6 倍）

**组件 2：Base Masked Transformer**（负责第 1 层 token）
- 训练时：随机把 40-80% 的 token 用 `[MASK]` 替换，网络预测它们原来是什么（cosine schedule：训练初期 mask 比例较高约 80%，后期递减到约 40%，使训练逐步从易到难）
- 推理时：从全 `[MASK]` 开始，每步预测所有被 mask 的位置，保留高置信度的，其余重新 mask，**10 步填完**

**组件 3：Residual Masked Transformer**（负责第 2-6 层 token）
- 基于已填好的第 1 层，并行预测第 2-6 层所有 token（**5 步**，每步一层）

![MoMask完整三段流水线示意图](/images/Text-to-Motion-Generation-DeepDive/fig7.png)

*图 7：MoMask 的完整三段流水线。输入文本 → RVQ 6 层码本的第 1 层由 base transformer 10 步迭代填出 → residual transformer 5 步并行补齐第 2-6 层 → RVQ decoder 合成回 motion。相比 MDM 的 1000 步 diffusion 和 T2M-GPT 的 196 步自回归，MoMask 只要 15 步并行就能完成。*

### 4.3 Masked Modeling 和 BERT 的相同与不同

**相同**：
- 训练目标都是"给 masked 位置预测原 token"
- 都有 cross-entropy loss
- 都是 bidirectional attention（而非 causal）

**不同**（MoMask 独有）：
- BERT 训练时 mask 比例固定 15%；**MoMask 用 40%-80% 的 cosine schedule**（为了匹配推理时从高 mask 到低 mask 的分布）
- BERT 是判别任务；**MoMask 是生成任务**（每步填一批 + 下步继续填 + 最终生成完整序列）
- BERT 一次性输入全部；**MoMask 的推理是迭代式的**（类似扩散的 step）

![MoMask训练与推理过程示意图](/images/Text-to-Motion-Generation-DeepDive/fig8.png)

*图 8：MoMask 训练时随机 mask 40-80% 的 token，网络预测被 mask 位置的原 token。推理时从全 mask 开始，每一步预测所有位置，保留高置信度的 token，剩下的继续 mask，10 步后第 1 层填满。*

### 4.4 Hand-calc 4 —— MDM vs MoMask 的加速比 🔢

**前提**：两者每步 Transformer 前向计算量相当（都对完整序列做一次 attention）。

**MDM**：1000 步 diffusion
**MoMask**：15 步（10 + 5）

**加速比**：1000 / 15 ≈ **66.7 倍**

**质量呢？**（来自原论文 HumanML3D benchmark）：
- MDM FID = 0.544
- MoMask FID = 0.045（越低越好）

**MoMask 不但快 66 倍，FID 还更低**——这是工程上的全胜。

![三条路推理时间对比图](/images/Text-to-Motion-Generation-DeepDive/fig9.png)

*图 9：三条路的推理时间对比。纵轴 log scale 下 MoMask 比 MDM 快两个数量级，且 FID 指标更好。*

### 4.5 MoMask 的优缺点

**优点**：
- **快**（0.15 秒）
- **质量好**（FID 最低）
- **可控性强**（可以只 mask 想改的部分，保留其余）

**缺点**：
- **训练 pipeline 复杂**（要训 RVQ + base + residual 三个，且相互依赖）
- **代码量大**（~2000 行 PyTorch，是 T2M-GPT 的 3 倍）
- **调参门槛高**（mask schedule、置信度阈值、cfg scale 都要调）

### 4.6 章末自测

<details>
<summary>如果把 MoMask 的迭代步数从 15 增到 50，质量会提高吗？</summary>

**极小提升**（已经接近饱和）。这和 MDM 的 50-vs-1000 不同——MoMask 因为 RVQ 给了它一个"清晰起点"（离散 token），前 10 步就填得差不多了，再加步数只是微调。

</details>

<details>
<summary>为什么 MoMask 要 RVQ 而不是普通 VQ（像 T2M-GPT）？</summary>

普通 VQ 只有一层 token，信息瓶颈太严。RVQ 第 1 层捕大动作、后几层捕细节，**解码时把 6 层相加**，总容量 = 6×512 = 3072 个"动作词"。这让 MoMask 的重建误差明显小于 T2M-GPT。

</details>

### 5.2 TA 实战决策树

**场景 1：快速做 demo 看效果** → **T2M-GPT**
- 模型小、训练成熟、推理足够快（1-2 秒可接受）
- HuggingFace 有 pretrained checkpoint，开箱即用

**场景 2：做高质量的 showcase 视频** → **MoMask**
- FID 最低 = 视觉质量最好
- 推理 0.15 秒意味着可以一晚上批量生成 1000 段挑选

**场景 3：做动画编辑工具** → **MDM**
- 独家支持 inbetween / 局部 edit
- 用户可以只改某一段，其余保持
- 推理慢但编辑场景允许用户等 1 分钟

**场景 4：NPC 实时响应** → 三条都不完全够
- MoMask 0.15 秒最接近，但仍属于"触发后立刻播放"而非"完全实时"
- 要做真正实时 → Phase 3 的 Physics-RL 路线（DeepMimic / AMP）

## §6 工程现实：CLIP、CFG 调参、263 维到 UE [L1 扩展]

### 6.1 CLIP 文本 encoder 的一句话介绍

三条路的 text encoder 几乎都是 **CLIP ViT-B/32 的 text tower**：
- 输入：任意英文句子（≤ 77 token）
- 输出：512 维 context 向量
- 训练来源：4 亿图文对对比学习（Radford et al., ICML 2021）
- **为什么用 CLIP 而不是 BERT**：CLIP 的文本 embedding 天然和视觉/动作语义对齐（它见过"跑步的人"图 + 文本的大量配对），迁移到 motion 比 BERT 好。

### 6.2 把 CLIP 向量塞进三条路的具体方式

| 方法 | 插入方式 |
|---|---|
| T2M-GPT | 512-dim → linear → prepend 到 motion token 序列前 |
| MDM | 512-dim → project → prepend 到 (T=196) 序列前（作为条件 token） |
| MoMask | 512-dim → project → 同上 prepend（base 和 residual 各有一套） |

**共同点**：都用 self-attention 让 motion 每个位置都能"看到"文本，不用 cross-attention。

### 6.3 CFG scale 调参实测（只适用 MDM / MoMask）

| scale w | 效果 | 推荐场景 |
|---|---|---|
| 1.0 | 基线，多样性最好 | 探索式生成（想看各种可能） |
| 2.5 | 默认，平衡 | 标准生成（demo） |
| 5.0 | 贴文字更强，多样性损失 ~30% | 文字描述精确的场景 |
| 7.5 | 极端贴文字 | 对比实验，不建议上线 |

### 6.4 从 263 维到 UE 骨骼的 retarget

**263 维 → UE5 骨骼**的详细步骤：

1. **坐标系转换**：
   - HumanML3D 是 Y-up，UE5 是 Z-up
   - 转换公式：`(x, y, z)_humanml3d → (x, z, y)_ue5`
   - 注意：旋转四元数也要相应转换——Y-up 到 Z-up 需要绕 X 轴旋转 +90°（右手定则，拇指朝 +X 方向，四指从 Y 转向 Z），对应四元数（w,x,y,z）= (√2/2, √2/2, 0, 0) ≈ (0.707, 0.707, 0, 0)；原始坐标乘以该四元数即可

2. **提取关节位置**（前 63 维 = 21 关节 × 3D）：
   - 关节顺序：root, pelvis, spine1, spine2, spine3, neck, head, left_hip, left_knee, left_ankle, right_hip, right_knee, right_ankle, left_shoulder, left_elbow, left_wrist, right_shoulder, right_elbow, right_wrist, left_toe, right_toe
   - 注意：HumanML3D 的 root 是世界坐标，需要转换成 UE5 的 Character 局部空间

3. **Retarget 映射到 UE5 骨架**：
   - 如果是 MetaHuman：用 ML Deformer 或 Custom Rig
   - 如果是普通骨架：写一个简单的 RemapTable（JSON），定义 HumanML3D 关节 → UE5 骨骼的映射
   - 示例映射：`"pelvis" → "pelvis_01"`，`"left_knee" → "thigh_l_01"`

4. **位置 → 旋转（FK 求解）**：
   - 用 Laurent's Inverse Kinematics 或 UE5 的 `FABRIK` solver
   - 或者直接用 6D 旋转（第 64-189 维）转换成 FQuat，跳过 FK

5. **处理 Root Motion**（前 4 维）：
   - 第 1-2 维：root velocity (x, z)
   - 第 3 维：root height velocity (y)
   - 第 4 维：root angular velocity (yaw)
   - 在 UE5 里用 `UCharacterMovementComponent::AddInputVector()` 或自定义 `AActor::AddActorWorldOffset()`

6. **脚接触修正**（第 260-263 维）：
   - 如果 `foot_contact[left] = 1`，冻结 left_ankle 的 world position
   - 用 UE5 的 `FBIK`（Full Body IK）或 simple copy rotation 避免脚滑

**坑点**（实战）：
- HumanML3D 的 Y-up vs UE Z-up 坐标系差异要先 flip
- Root motion 一定要单独处理（前 4 维）否则角色会"原地踏步"
- 脚接触标签（4 维）用来做 Foot IK 修正，避免穿地

![文字到UE5骨骼动画完整管线图](/images/Text-to-Motion-Generation-DeepDive/fig11.png)

*图 11：从文字到 UE5 骨骼动画的完整管线。文字 → CLIP → motion 生成模型 → 263 维 HumanML3D 向量 → retarget 到 UE5 骨骼 → Animation Asset。*

### 6.5 章末自测

<details>
<summary>CFG scale 调到 10 生成的动作看起来"僵硬夸张"，为什么？</summary>

因为 `ε_final = ε(∅) + w·[ε(c)-ε(∅)]`，w=10 把"文字方向"放大得远超"自然分布"，生成的动作会"过拟合文字"而失去自然。**这是 CFG 的通用规律**，图像扩散里也一样。

</details>

## §7 下节课预告：从生成到交互 [L0 核心]

本文三条路都是**生成式 ML 路线**：给文字 → 吐 motion。

但动画工业另一条根深蒂固的路是 **PFNN（Phase-Functioned Neural Networks）**，它不需要文字条件，而是**基于当前状态（phase + trajectory）预测下一帧**。这是 2017 年 Holden 的论文，很多商业游戏（《荒野大镖客 2》的马匹、《看门狗：军团》的 NPC）背后都是类似思路。

**学习路线衔接**：
- **今天同日的另一篇**（`2026-04-25-DeepMimic-AMP`）讲的是物理强化学习路线（DeepMimic / AMP），它是"让角色在物理仿真里学动作"，和本文的"数据驱动生成"是互补的两种思路
- **明天**（`2026-04-26`）有两条线：MDM/MLD 的扩散深讲（本文 §3 的延伸），和 PFNN 的相位函数深讲
- PFNN 的相位概念会贯穿后续 ASE（`2026-04-27`）、DeepPhase（`2026-04-29`）、Learned MM（`2026-04-28`）等多篇文档

**学完 PFNN 你能做什么**：在 UE 里用自己的动捕数据训一个 PFNN 角色控制器，实时响应操控。

## 📎 附录 A：评估指标 FID 和 R-Precision 从零讲起 [L1 扩展]

### A.1 FID（Frechet Inception Distance）

**直觉**：
> "生成的动作分布和真实动作分布有多像？"

**具体做法**：
1. 取一个预训练的 motion encoder（类似图像的 InceptionV3）
2. 把真实 motion 和生成 motion 分别喂进去，得到两组特征向量
3. 把两组特征各自拟合成一个多元高斯分布 `N(μ_real, Σ_real)` 和 `N(μ_gen, Σ_gen)`
4. 计算两个高斯分布之间的 Frechet 距离：
   $$\text{FID} = \|\mu_{\text{real}} - \mu_{\text{gen}}\|^2 + \text{Tr}(\Sigma_{\text{real}} + \Sigma_{\text{gen}} - 2(\Sigma_{\text{real}} \Sigma_{\text{gen}})^{1/2})$$

**越低越好**。典型值：
- 真实-vs-真实 ≈ 0.002（ground truth 上限）
- MoMask ≈ 0.045（当前 SOTA）
- MDM ≈ 0.544
- T2M-GPT ≈ 0.116

**FID 的缺点**：
- 对 encoder 敏感（不同 encoder 会给不同 FID）
- 反映分布整体，**不反映单样本好坏**

### A.2 R-Precision

**直觉**：
> "给定生成动作 + 一组候选文字（真实那句 + 31 句干扰项），生成的动作能不能匹配到真实那句？"

**具体**：
- 把生成 motion 和 32 句文字（1 真 + 31 假）分别过 CLIP-like encoder
- 计算 motion 和 32 句的相似度排名
- **Top-k R-Precision** = 真实那句排在前 k 的概率

**越高越好**。典型值：
- Ground Truth Top-3 ≈ 0.850
- MoMask ≈ 0.807
- T2M-GPT ≈ 0.775

**R-Precision 的优点**：
- 直接衡量"生成是否符合文字描述"
- 比 FID 更贴近用户感知

### A.3 为什么两个指标都要看？

- 只看 FID：可能"生成的动作整体像真实分布但不贴文字"
- 只看 R-Precision：可能"贴文字但动作不自然（崩塌、抖动）"
- 两个一起看才能判断生成质量

## §1 把 DDPM 直接搬到动作上，会卡在哪？ [L0 Core]

### 1.1 你周一上午接到的需求

老板拍肩膀："Stable Diffusion 现在文生图很稳，咱们做个文生动作的 demo 给客户演示，下周一要看到东西。" 你打开 [DDPM 论文](https://arxiv.org/abs/2006.11239)，复制了一份图像扩散代码，把"3 通道 256×256 图"换成"263 维 × 196 帧动作"，跑了一晚上。

第二天早上一看：**生成的动作能动，但脚在地板下面打滑、左手腕到肩膀的距离一秒里从 30cm 漂到 45cm 又漂回 30cm**。客户演示翻车。

这一节回答：**为什么这个翻车不是 bug，而是 DDPM 默认配置在动作数据上的本质失败**？

### 1.2 图像扩散为什么"不需要"想这件事

先回到图像。一张 256×256×3 的图像，每个像素的 R/G/B 之间没有"必须满足的几何约束"——你可以把红色通道整体亮 20%，整张图还是一张有效图像。换句话说，**图像的有效空间是连续且开放的**。

DDPM 在这种数据上的训练目标极其简单（4/23 笔记 §3 已推导）：

```
L_simple = E_t,x_0,ε [|| ε - ε_θ(x_t, t) ||²]
```

网络只需要学"输入带噪图像和时间步 t，输出加进去的噪声 ε"。这个 ε 是一个 256×256×3 的张量，与 x_0 同形。**只要这个 ε 预测准了，每个像素自动落在合理区间，没人会去检查"红色像素之间是否满足某条几何方程"。**

### 1.3 动作数据的"硬约束"

动作数据完全不是这样。一帧人体姿势（HumanML3D 的 263 维表征）里藏着一堆**硬约束**：

1. **关节长度恒定**：上臂在第 1 帧和第 100 帧必须长度一样（30cm 就一直 30cm，骨头不会缩）。
2. **脚不能穿地板**：当 foot_contact 标志为 1 时，那只脚的位置应该被钉在地上。
3. **关节角度有范围**：膝关节不能反向弯到 200°。
4. **速度连续**：相邻两帧关节位置差不能突然跳到 1m（除非角色被打飞）。

这些约束不存在于图像里。**而 DDPM 的 ε 预测模式有一个致命特点：网络在训练时只看到加噪后的 x_t，从来不"看见"它正在帮哪个 x_0 还原。**

具体一点。前向扩散的关系是：

```
x_t = sqrt(α̅_t) · x_0  +  sqrt(1-α̅_t) · ε
```

这是 4/23 笔记 §2 已经熟到不行的公式。给一个具体的数：HumanML3D 默认 T=1000，t=500 时 α̅_500 ≈ 0.005。代入：

```
x_500 ≈ sqrt(0.005) · x_0  +  sqrt(0.995) · ε
       ≈ 0.0707 · x_0     +   0.9975 · ε
```

**0.07 倍的原始动作 + 0.9975 倍的纯噪声**——这一帧从视觉上看就是一团雪花，几乎全是 ε。网络在 ε 预测模式下，是在学"输入这团雪花，吐出里面那 0.9975 倍的 ε"。整个训练过程，**网络从未被强制问过"你恢复出来的 x̂_0 满足那一堆几何约束吗"**。

### 1.4 关键配对：ε 预测 vs x_0 预测，差异是什么

这是本节最重要的一句话，必须能背出来：

> **ε 预测让你在每一步只看噪声；x_0 预测让你在每一步都能看到一个"模型当前以为的干净动作"——而几何约束只能定义在干净动作上。**

| 预测目标 | 网络输出 | 每步能算几何 loss 吗 | 数学等价吗 |
|---|---|---|---|
| **ε 预测** | ε̂_θ(x_t, t)，与 x_t 同形的噪声估计 | **不能直接算**——要先反推 x̂_0 = (x_t − sqrt(1-α̅)·ε̂)/sqrt(α̅)，t 大时 sqrt(α̅) 极小，反推数值极不稳 | 与 x_0 预测在 L_simple 上数学等价（只是 weighting 不同） |
| **x_0 预测** | x̂_0_θ(x_t, t)，与 x_0 同形的"干净动作猜测" | **可以直接算**——FK 一下、算关节位置 / 速度 / 接触 | 同上 |

第三列就是 MDM 整篇论文的核心动机。Tevet 等人 2022 的核心贡献不是"换了个网络"，而是**换了个预测目标，从而打开了"在每个扩散步上加几何 loss"这扇门**[S1]。

### 1.5 hand-calc 锚点：t=500 这一帧实际长什么样

让我们把 §1.3 的数字落到一个具体的 walk pose 上。HumanML3D 一帧 263 维，挑出"右脚 Y 坐标"这一维（位置维度，单位 m）。

- 干净 x_0：右脚 Y = 0.000（站立时贴地）
- 加 t=500 步噪声后：x_500 的右脚 Y = 0.0707 × 0.000 + 0.9975 × ε，ε ~ N(0,1)
- 假设这次采到 ε = 0.8，则 x_500 的右脚 Y ≈ 0.798 m

**这意味着 t=500 时，"右脚"在张量里漂到了 80cm 高的位置**。如果网络此时只学"预测 ε"，它根本不需要去想"我帮还原出来的右脚是不是落在地上"——它只在做一个像素级噪声拟合。等到全部 1000 步反推完，几何错误已经累积成了演示翻车那个画面。

### 1.6 这一节的结论

DDPM 的默认 ε 预测在图像上够用，是因为图像没有硬几何约束。**动作有硬几何约束，所以你必须换成 x_0 预测，并且在每步加几何 loss。** §2 讲怎么换预测目标，§3 讲那三个几何 loss 长什么样。

## §2 MDM 的回答（一）：把预测目标从 ε 换成 x_0 [L0 Core]

### 2.1 你现在要写的训练循环长什么样

延续 §1 的剧情。你现在拍板："换 x_0 预测。"打开 PyTorch，要改的就一行：

```python
# 旧：DDPM 默认
loss = F.mse_loss(eps_pred, eps_target)

# 新：MDM 默认
loss = F.mse_loss(x0_pred, x0_target)   # x0_target 就是原始干净动作
```

听起来轻飘飘，但要确认这件事**在数学上不会让训练崩**——下面用一段把它讲清楚。

### 2.2 为什么换目标后训练还能 work——一句话理由

回到 4/23 笔记 §3.5 的关键事实：DDPM 的简化损失 L_simple 在"预测 ε"和"预测 x_0"之间是**数学等价的**，区别只在每个时间步 t 的权重 w(t)。证明只用一行：把 `x_t = sqrt(α̅_t)·x_0 + sqrt(1-α̅_t)·ε` 代入 ε = (x_t − sqrt(α̅_t)·x_0) / sqrt(1−α̅_t)，两个 MSE 之间差一个常数缩放。

所以**把预测目标从 ε 换到 x_0，原则上不会改训练的"能不能收敛"**，只会改"每个 t 在总 loss 里的有效权重"。这里需要精确一下：原始 DDPM 的 L_simple 是**加权平均**，权重 ~ β_t²/σ_t，对大 t（高噪声）给更大权重；MDM 改用 x_0 预测后，采用 `L_simple_x0 = E[||x_0 − x̂_0(x_t, t)||²]` 这个**等权形式**[S1]，即所有 t 的权重都设为 1。这个选择有一个隐含偏置：**对早期（t 小）的恢复给的权重相对更大**——而早期恢复正是几何约束最敏感的时期，正好对上号。

### 2.3 关键配对：ε_θ 输出层 vs x̂_0_θ 输出层，长什么样

| 项目 | ε 预测网络 | x_0 预测网络 |
|---|---|---|
| 输入 | (x_t, t, condition) | 同左 |
| 输出张量 shape | 与 x_0 同形：[batch, frames, 263] | **完全一样**：[batch, frames, 263] |
| 输出含义 | 张量每个分量是噪声估计 | 张量每个分量是干净动作分量估计 |
| 计算几何量 | 需先反推 x̂_0，t 大时数值爆 | **直接拿输出做 FK / 算速度 / 算接触** |
| L_simple 损失值 | 与目标 ε 比较 | 与目标 x_0 比较 |

形状一样、参数量一样、Transformer Encoder 一样。**唯一变化是损失函数的 target 和"网络输出在反向传播图里下游能怎么用"这两件事。**

### 2.4 MDM 的网络长什么样

为了让你脑里有个具体的东西可以指——MDM 用的是 8 层 Transformer Encoder，没有任何 U-Net 或 Conv1D：

```
输入序列：[CLS_t  CLS_cond  frame_1  frame_2  ...  frame_196]
                ↓                  ↓
         t 的 sinusoidal     CLIP 文本 embedding
         positional encoding (512 维)
                ↓
         8 × Transformer Encoder Layer (d_model=512, nhead=4, ff=1024)
                ↓
         去掉前两个 CLS token，剩下 196 帧 × 512 → Linear → 196 帧 × 263
                ↓
         x̂_0 ：与原始动作同形的干净估计
```

几个要点：

1. **t 走 CLS 位**：和 ViT 把 image class 放 CLS 位一样，时间步 t 编码成一个 token 加在序列首位。
2. **文本走 CLS 位**：CLIP-ViT-B/32 的 [EOS] token 嵌入（512 维）作为另一个 CLS。
3. **没有 U-Net**：因为动作没有 2D 局部性，全连接的自注意力比卷积合适。
4. **参数量 ~45M**：比 Stable Diffusion 的 ~860M U-Net 小一个数量级——动作序列短，模型可以小。

### 2.5 hand-calc 锚点：跑一次前向，看输出该是什么

构造一个能心算的小例子。简化到 5 帧 × 4 维（实际 196 帧 × 263 维），干净动作 x_0：

```
x_0 = [[0.0, 0.0, 0.0, 0.0],   # 第 1 帧站立
       [0.1, 0.0, 0.0, 0.0],   # 第 2 帧右脚抬起 0.1
       [0.2, 0.0, 0.0, 0.0],   # 第 3 帧 0.2
       [0.1, 0.0, 0.0, 0.0],   # 第 4 帧落下
       [0.0, 0.0, 0.0, 0.0]]   # 第 5 帧站立
```

t = 200，假设 α̅_200 ≈ 0.4，sqrt(α̅) ≈ 0.632，sqrt(1-α̅) ≈ 0.775。采一组高斯噪声 ε（每个分量独立 N(0,1)），构造 x_200：

```
x_200[i,j] = 0.632 · x_0[i,j] + 0.775 · ε[i,j]
```

把 x_200 和 t=200、文本 condition "a person walks forward" 喂进网络，**网络应该输出 x̂_0 ≈ x_0**——也就是上面那个 5×4 矩阵。损失是 `mean((x̂_0 - x_0)²)`。如果网络只学到"输出 x_t 本身"，损失会大约是 0.775² × Var(ε) ≈ 0.60；学得好的网络应该把损失压到 0.01 量级。

**记住一件事**：训练时网络一次性吃整段 196 帧 + 一个标量 t + 一段文本，吐回整段 196 帧的 x̂_0。**不是逐帧预测，也不是自回归**。这一点和 §3 之后讲 T2M-GPT 自回归路线截然不同。

### 2.6 这一节的结论

预测目标换成 x_0，训练循环改一行；网络结构不变；唯一打开的新能力是"输出可以直接做 FK 算几何"。下一节就用这个能力。

## §3 MDM 的回答（二）：几何三件套损失 [L0 Core]

### 3.1 几何 loss 到底要约束什么

§2 我们打开了"在每步上能直接看到 x̂_0"这个能力。但能看到不代表会用。这一节要回答：**给你一个完整的 x̂_0（一段 196 帧 × 263 维的干净动作猜测），你应该往它身上加哪些约束，才能压住 §1 那个翻车 demo 的脚滑、关节漂、抖动？**

MDM 论文给出三个，覆盖了 95% 的可视化痛点[S1]：

1. **关节位置 loss（L_pos）**：你猜出来的关节位置和真值是否一致——压"漂"
2. **关节速度 loss（L_vel）**：相邻两帧关节位置差是否符合——压"抖"
3. **脚部接触 loss（L_foot）**：foot_contact 标志为 1 时脚的速度应近 0——压"滑步"

总损失是这三个 + 主损失的加权和：

```
L_total = L_simple  +  λ_pos · L_pos  +  λ_vel · L_vel  +  λ_foot · L_foot
```

MDM 默认 λ ≈ 1.0 量级（HumanML3D 配置）[S1]。

### 3.2 关键配对：L_pos / L_vel / L_foot 各管什么——一句话

> **L_pos 管"位置错"，L_vel 管"节奏错"，L_foot 管"滑步错"。**

| Loss | 数据形式 | 痛点对应 | 计算 |
|---|---|---|---|
| **L_pos** | 22 个关节的 3D 世界坐标（22×3=66 维） | 关节漂、骨头长度变 | FK(x̂_0) 与 FK(x_0) 的逐关节 MSE |
| **L_vel** | 相邻两帧关节坐标差 | 高频抖动、突然跳变 | (FK_t − FK_{t-1}) 的逐关节 MSE |
| **L_foot** | 双脚的 3D 速度 × 接触掩码 | 接地脚滑步 | mask · ‖foot_vel‖² 的均值 |

### 3.3 L_pos —— 让位置回到正确的几何位置

> **为什么不能直接用 263 维里的"关节局部位置"（63 维）来算 L_pos？**
> HumanML3D 的 263 维里确实有 joint local positions（21 关节 × 3 = 63 维），但那是**相对于父关节的局部坐标**。几何约束（脚不能穿地、关节长度恒定）是定义在**世界坐标系**下的。比如"脚 Y=0"是地面接触约束，在 local space 里这个值随根关节姿态变化，不是常数。所以必须做 FK，把局部旋转 + 根位移转成世界坐标下的关节位置，才能和真值比较。

HumanML3D 的 263 维表征里，已经有"关节局部位置"这一项（local pos 63 维）。但**真正决定"看起来对不对"的是世界坐标系下的关节位置**——也就是要先做 FK（Forward Kinematics，正向运动学）：

```
joint_world = FK(root_translation, root_rotation, joint_local_rotations)
```

22 个 SMPL 关节，每个 3 维坐标，总共 66 维。L_pos 就是网络输出做 FK 后与真值做 FK 后的逐关节 MSE：

```
L_pos = mean_{帧, 关节} || FK(x̂_0)[joint] − FK(x_0)[joint] ||²
```

这一项会强迫网络去**学习满足关节长度恒定**——因为只要某一帧网络输出了"上臂 25cm"而真值是"上臂 30cm"，FK 出的肘部位置就错了，L_pos 立刻报警。

### 3.4 L_vel —— 让节奏不抖

L_pos 只看每一帧的"位置对不对"，看不到"两帧之间过渡得自不自然"。所以再加一项"速度 loss"：

```
L_vel = mean_{帧, 关节} || (FK(x̂_0)[t] − FK(x̂_0)[t−1])  −  (FK(x_0)[t] − FK(x_0)[t−1]) ||²
```

注意：**比较的不是位置，是位置差**。这一项专门压"高频抖动"——网络如果在第 50 帧把右手位置预测对了，但在第 51 帧手位置突然跳一下，L_vel 立刻报警。

### 3.5 L_foot —— 接地脚不能滑

最直观也最容易解释的一项。HumanML3D 数据集里给每一帧标了双脚的接触状态（contact ∈ {0, 1}）。如果某一帧 contact=1，那只脚此时应该**钉在地上**——速度近似 0。

```
L_foot = mean_{帧} contact_mask[t] · || foot_vel[t] ||²
```

这就是论文 Eq. 9。它只在 contact=1 的帧上罚速度，在腾空帧上不罚。

### 3.6 hand-calc 锚点：foot loss 一段 5 帧实算

构造一段简化数据，让你能心算每一项。一只右脚，5 帧速度（米/秒）和接触标志：

```
foot_vel  = [0.02, 0.03, 0.50, 0.04, 0.02]
mask      = [   1,    1,    1,    1,    1]   # 5 帧都接触地面
```

第 3 帧的 0.50 是个典型滑步——理论上脚钉地不该有速度。

逐项算：

```
mask[t] · vel[t]² = [0.0004, 0.0009, 0.2500, 0.0016, 0.0004]
sum = 0.2533
L_foot = sum / 5 = 0.05066
```

**0.05 这个数主要被第 3 帧贡献**（0.25 / 0.2533 ≈ 99%）。也就是说，只要某一帧出现"接触脚却在动"，loss 立刻被第 3 帧拉高，反向传播会精准压回那一帧。这就是"硬几何约束被显式编码"的样子。

把 mask 改成全 0（脚全程腾空），L_foot = 0——网络此时不被罚滑步，因为没有接触。

### 3.7 总损失实算

把 §1.3 的 t=500 时 loss 配置代进来。一段 196 帧、22 关节的训练样本：

- L_simple = 0.06（从 263 维 MSE 算）
- L_pos = 0.04（22 关节 × 196 帧 FK 后 MSE）
- L_vel = 0.02（位置差 MSE）
- L_foot = 0.05（接触脚的速度平方）

总损失（λ 全 = 1.0）：L_total = 0.06 + 0.04 + 0.02 + 0.05 = 0.17。**几何三项加起来贡献了总损失的 65%**——这正是 MDM 比 ε 预测路线在动作上视觉质量好得多的根本原因。

### 3.8 工程要点

1. **FK 必须可微**：所有 L_pos / L_vel 都要反向传播，FK 实现要用 PyTorch 张量自动微分。SMPL 官方 PyTorch 实现就是可微的。
2. **λ 调参**：λ 调太大会让网络过分追求几何精度，文本对齐变差（动作和描述对不上）；调太小则脚滑回来。MDM 默认 1.0 在 HumanML3D 上 work，自己数据集要重新扫一遍。

> **λ 调整经验范围**（在 HumanML3D 上实验得出的参考值）：
> | λ 值 | 几何精度 | 文本贴合（R-Precision） | 适用场景 |
> |---|---|---|---|
> | **0.1-0.5** | 弱，脚滑明显 | 高 | 文本驱动优先，几何要求低 |
> | **1.0（默认）** | 中等，脚滑可接受 | 中等 | 通用场景 |
> | **2.0-5.0** | 强，脚滑很少 | 下降 5-10% | 物理交互场景（地形、刚体） |
> | **> 5.0** | 过强，动作僵硬 | 显著下降 | 不推荐，改用 OmniControl 等控关节方法 |

3. **几何 loss 只在 t < 某阈值时启用**：t 大时 x̂_0 还很糊，几何 loss 信号噪声太大。MDM 实现中，L_pos / L_vel / L_foot 整个训练周期都开（论文未做这个 trick），但 PriorMDM[S12] 等后续工作发现"只在 t < 500 启用几何 loss"会更稳。

<summary>3.9 自检：如果你拿到一段生成动作发现脚有微微滑步，应该调哪个 λ？</summary>

直接调大 λ_foot（默认 1.0 → 试 2.0 或 5.0）。但要小心同时观察 L_simple 和 R-Precision 是否退化——如果文本贴合度掉太多，说明几何项压制了语义学习，需要回调或同时增大数据增广。

## §4 MDM 的代价：为什么一段动作要跑十几秒 [L0 Core]

### 4.1 你周三下午发现的"性能问题"

§3 的 MDM 训练终于跑通，几何也对了，客户演示也救回来了。但一周后产品同学找你："这个生成 5 秒动作要 12 秒，**做不了实时交互**啊。" 你打开计时器，发现确实——HumanML3D 标准配置下，MDM 用 DDPM 1000 步采样在单卡 A100 上一段 196 帧动作 **约 12.8 秒**[S2]，DDIM 50 步加速到 **约 0.6-1 秒**，仍然不够实时。

这一节回答：**慢在哪？哪些地方还可能挤压？**

### 4.2 一次完整推理在做什么

把"生成一段 5 秒动作"拆开看：

```
循环 T 步（DDPM 1000 / DDIM 50）：
   1. 当前 x_t（196 帧 × 263 维 = 51548 维张量）
   2. 喂进 8 层 Transformer（45M 参数）做一次 forward
   3. 拿到 x̂_0（同形 51548 维）
   4. 算 x_{t-1} = function(x̂_0, x_t, t, noise)
```

**核心成本 = 步数 × 单步 forward 时间**。两个因子都很贵。

### 4.3 单步 forward 的成本拆解

Transformer Encoder 的复杂度公式：

```
单层 attention 复杂度 ≈ O(L² · d_model)
单层 FFN 复杂度       ≈ O(L · d_model · d_ff)
```

代入 MDM 配置：L=196 帧, d_model=512, d_ff=1024, 8 层。

```
单层 attention：198² · 512 ≈ 2.00 × 10⁷ FLOPs
单层 FFN：       198 · 512 · 1024 ≈ 1.03 × 10⁸ FLOPs
单层合计 ≈ 1.22 × 10⁸ FLOPs
8 层合计 ≈ 9.78 × 10⁸ FLOPs ≈ ~1 GFLOPs / step
```

A100 算力 ~312 TFLOPs（FP16），理论 1 GFLOPs / step 应该 < 1ms，但**实测 ~12ms / step**。差异来自访存、kernel 启动、batch=1 时矩阵乘没填满 SM。

### 4.4 关键配对：DDPM 1000 步 vs DDIM 50 步——差异是什么

> **DDPM 1000 步是按 Markov 链严格采样；DDIM 50 步是用确定性 ODE 路径"跳着"走。两者最终样本质量差异通常 < 5% FID，但时间差 20×。**

| 采样器 | 步数 | 路径 | A100 单段时间 | FID 变化 |
|---|---|---|---|---|
| DDPM | 1000 | Markov 概率链 | ~12.8 s | 基线 |
| DDIM | 50 | 确定性 ODE | ~0.6-1 s | +0.02 量级（基本无感） |
| DPM-Solver-2 | 20 | 高阶 ODE 求解器 | ~0.25 s | +0.01~0.05 量级 |

**实践上 MDM 默认推理一律用 DDIM 50，DDPM 1000 只在训练时用**。这是原 DDPM 论文 + DDIM 论文已证明的免费午餐[S3][S9]。即便如此，DDIM 50 步的 0.6 秒**仍然偏慢**——这就是 MLD 出场的动机。

> **DPM-Solver / DPM-Solver-2 说明**：这是 NeurIPS 2022 提出的高阶 ODE 求解器，可以用更少步数达到同等质量。上表 ~0.25s 是 MDM（raw motion）上 DPM-Solver-2 20 步的时间；MLD 上因单步成本更低，20 步约 0.1s 量级。详见 [DPM-Solver 论文](https://arxiv.org/abs/2206.00927)。

### 4.5 hand-calc 锚点：196 × 263 = 51548 维到底有多大

把"扩散张量大小"和"图像扩散"做个直观对比：

| 数据 | 形状 | 总维度 |
|---|---|---|
| Stable Diffusion 1.5 latent | 64 × 64 × 4 | 16384 |
| MDM 一段动作 raw | 196 × 263 | **51548** |

**MDM 的扩散张量比 SD 的 latent 大 3.1 倍**——而且 SD 的 U-Net 用了大量 batch 和 spatial parallel，MDM 的 Transformer 是序列长 196，每步 attention 都要算 196²=38416 个对比对。**这是 MDM 慢的本质原因：你在原始动作维度上跑扩散，等价于在"未压缩"的动作上做 50-1000 次重活**。

如果能把 51548 维压到 256 维（200×压缩），单步成本下降 ~200×，总时间从 0.6s 降到 0.003s 量级。MLD 就是这条路。

### 4.6 这一节的结论

MDM 慢，慢在"扩散直接跑在原始 196 × 263 维动作上"。即便用 DDIM 50 步加速，单段仍要 0.6 秒。**要拿到实时（< 100ms），必须把扩散搬到一个低维 latent 空间——这正是 MLD 的核心改动**。

![fig7](/images/MDM-MLD-MotionDiffusion-DeepDive/fig7.png)

<details>
<summary>4.7 自检：如果只是把 MDM 的层数从 8 减到 4，能加速到 100ms 以内吗？</summary>

不能。单步 forward 成本和层数线性，减半只能从 0.6s 降到 0.3s；但样本质量会明显下降（论文消融实验中 4 层比 8 层 FID 退化 ~30%）。**要拿到 10-100× 的加速，必须改"扩散张量本身的维度"，而不是改网络深度**。这是 latent diffusion 的本质优越性。
</details>

## §5 MLD 的回答：把扩散搬到 latent 空间 [L0 Core]

### 5.1 你周四上午看到的论文

产品同学下了通牒"周一前要 100ms 以内"。你打开 arXiv，看到 2022 年底的一篇 [MLD: Executing your Commands via Motion Diffusion in Latent Space](https://arxiv.org/abs/2212.04048)[S2]，副标题就两个字"latent"。

这一节回答：**把扩散从 raw motion 搬到 latent，要做什么？牺牲什么？为什么能快几十倍？**

### 5.2 思路从 Stable Diffusion 抄来

回忆 LDM（Latent Diffusion Model，CVPR 2022）[S5] 在图像上的做法：

```
图像 (256×256×3 = 196608 维)
   ↓ VAE encoder
latent (64×64×4 = 16384 维)
   ↓ Diffusion 在 latent 跑
latent_denoised
   ↓ VAE decoder
图像
```

12× 压缩，扩散网络只需在 16384 维上工作，速度大涨。

MLD 把这套搬到动作上：

```
动作 (196×263 = 51548 维)
   ↓ Motion VAE encoder
μ (256 维) + σ (256 维)            ← 200× 压缩！z ~ N(μ, σ)
   ↓ Latent Diffusion
latent_denoised
   ↓ Motion VAE decoder
动作
```

注意 MLD 默认 K=1（只 1 个 latent token），latent 维度 256。**整段 196 帧动作被压成 1 个 256 维向量**——这是相当激进的压缩，比 LDM 的 12× 高一个数量级。

> **K 是什么？** K 是 latent token 的数量。K=1 时 latent 是 1×256 维向量；K=7 时 latent 是 7×256 维矩阵（相当于把动作拆成 7 个语义段，每段一个 token）。K 越大，VAE 重建质量越高，但 Latent Diffusion 的单步成本从 O(1²) 涨到 O(K²)（attention 复杂度），推理时间也随之增加。MLD 论文实验：K=1 时 FID=0.473，K=7 时 FID=0.312（更好），但 50 步推理时间从 0.22s 涨到 ~0.45s。

### 5.3 关键配对：raw motion 上的扩散 vs latent 上的扩散

> **MDM 让扩散网络看见 51548 维"原始动作"；MLD 让扩散网络只看见 256 维"动作概要"。多出来的 Motion VAE 把"动作的几何细节"封装在编解码器里，扩散网络专注于学"语义到概要"的映射。**

| 项目 | MDM（raw） | MLD（latent） |
|---|---|---|
| 扩散张量维度 | 196 × 263 = 51548 | 1 × 256 = 256（默认 K=1） |
| 单步 forward 计算 | ~1 GFLOPs | ~5 MFLOPs（约 200× 少） |
| 几何 loss 怎么加 | 直接在每个扩散步加 | **加不上**——Latent Diffusion 拿不到关节坐标 |
| 几何约束放在哪 | 扩散步内 | **挪到 Motion VAE 训练阶段** |
| 50 步推理时间 | ~0.6 s | ~0.22 s（论文实测） |
| HumanML3D FID | 0.544 | 0.473（略好） |
| HumanML3D R-Prec Top-3 | 0.611 | 0.772（明显好） |

**两件令人意外的事**：

1. **MLD 不仅快，FID 还略好**——因为 256 维 latent 是 Motion VAE 学出来的"动作的有效流形"，扩散在这个干净流形上学得更稳。
2. **MLD 不能直接加几何 loss**——因为扩散步看不到关节坐标。几何约束被推到 Motion VAE 重建阶段一起学。

### 5.4 三阶段训练流程

MLD 是分阶段训练的，不像 MDM 端到端[S2]：

```
Stage 1：训 Motion VAE（独立训练）
  - encoder: 196×263 → 256 维 μ, σ
  - decoder: 256 → 196×263
  - Loss = L_recon + β · L_KL（VAE 标准损失，β ~ 1e-4）
  - 这个阶段顺便学会几何（在重建 loss 里隐含）

Stage 2：冻结 VAE，训 Latent Diffusion
  - 拿 Stage 1 的 encoder 把 dataset 全部预编码成 256 维 latent
  - 在 latent 上跑标准 DDPM，predict ε（这里又回到 ε 预测，因为 latent 没几何意义）
  - Loss = L_simple = E[||ε - ε_θ(z_t, t, condition)||²]
  - 用 CLIP-ViT-B/32 文本嵌入做 condition
  - **Condition drop = 10%**（和 MDM 一样），保证 CFG 在推理时可用

Stage 3（可选）：联合微调
  - 解冻 VAE，整体小学习率微调几个 epoch
```

### 5.5 hand-calc 锚点：加速比从哪来

精确算一下加速比，验证 §4.5 的"理论 200×"。

**MDM 单步成本**（§4.3）：~1 GFLOPs
**MLD 单步成本**（256 维输入）：

- 单层 attention：1² · 256 ≈ 256 FLOPs（K=1 时序列长 1，attention 退化）
- 单层 FFN：1 · 256 · 1024 ≈ 2.6 × 10⁵ FLOPs
- 单层合计 ≈ 2.6 × 10⁵ FLOPs
- 8 层合计 ≈ 2 × 10⁶ FLOPs ≈ 2 MFLOPs / step

**单步加速比**：1 GFLOPs / 2 MFLOPs ≈ **500×**。

但实际加速到了 0.6 / 0.22 ≈ 3× 而非 500×——为什么？因为：

1. **VAE 编解码本身有开销**：encoder + decoder 每次推理各一次 forward，加起来约 5ms。
2. **Latent Diffusion 用了同样的 8 层 Transformer**：虽然每步 forward 便宜，但 50 步 × 50ms 启动开销摊销不掉。
3. **K=1 时 attention 几乎退化**，但 FFN 还在转。

论文实测加速 ~58×（[S2] Table 4），把 K 调到 7（latent 容量更大）时加速比降到 ~30×，但 FID 反弹更小。**这是典型的"压缩比 vs 质量"的工程取舍**。

| K 值 | Latent 形状 | 50 步推理时间（A100） | HumanML3D FID | 适用场景 |
|---|---|---|---|---|
| **K=1** | 1×256 | ~0.22 s | 0.473 | 实时交互，短动作（≤5s） |
| **K=2** | 2×256 | ~0.25 s | 0.410 | 中等长度，质量稍好 |
| **K=4** | 4×256 | ~0.32 s | 0.351 | 长动作（5-10s） |
| **K=7** | 7×256 | ~0.45 s | 0.312 | 离线生成，质量优先 |

### 5.6 牺牲了什么

不能白吃午餐。MLD 相对 MDM 牺牲：

1. **几何 loss 不能加在扩散步**：必须靠 Motion VAE 把几何学进去。如果 VAE 重建质量不够好，最终输出会有微抖。
2. **K=1 时长序列容量有限**：196 帧压成 256 维 latent，长动作（>10 秒）重建会糊。可以加 K（K=2/4/7）但要重新训。
3. **三阶段训练流程更复杂**：MDM 端到端一个 loss 训完，MLD 要分别训 VAE → Diffusion，调试不那么直观。

### 5.7 关于"后面会讲"——CFG 在 latent 上怎么调

这里先记住一件事：**MLD 也用 CFG**（推理时把文本条件以 ~10% 概率丢成 ∅，推理时做 ε_uncond + s · (ε_cond - ε_uncond)）。MLD 论文用 s=7.5 是默认值，比 MDM 的 s=2.5 大一些。**为什么 MLD 要用更大的 s，以及 s 该怎么选**——延后到 §6，因为 §6 整章就是 CFG 调参手册，在那里集中讲更顺。这里只要先记住"MLD 也用 CFG，s 设大一点"即可。

<details>
<summary>5.8 自检：如果 Motion VAE Stage 1 训得很差（重建 FID 高），整个 MLD 会有什么症状？</summary>

最终生成动作会有"高频抖动 + 局部不连贯"。因为 Stage 2 的 Latent Diffusion 学到的 latent 即使完美还原，过 decoder 后也会被 VAE 的失真"涂上"。诊断方法：在 Stage 1 训完后单独跑 reconstruction（拿真实动作 encode→decode），看 reconstruction FID 是否 < 0.1；如果不达标，回去把 VAE 训稳再开 Stage 2。
</details>

## §6 CFG 在动作上的旋钮——s 调多少 [L0 Core]

### 6.1 用户反馈到了你手上

上线后第一波反馈：

- 用户 A："我说 'jump high'，结果生成了一个普通的小跳，**和文字描述差太远**。"
- 用户 B："我说 'walk slowly'，生成的动作虽然走得慢，但**僵得像机器人，没有人味**。"

这两类反馈都指向同一个旋钮：**Classifier-Free Guidance scale s**。这一节回答：**s 是什么，A 和 B 该往哪边调，怎么调**。

### 6.2 一句话先讲清 CFG 在干什么

回到 4/23 笔记 §5。CFG 在推理时做一件事：

```
ε̃(x_t, c) = ε_uncond(x_t)  +  s · (ε_cond(x_t, c)  −  ε_uncond(x_t))
```

把它读出来：**"无条件预测的噪声"加上"s 倍的'条件预测噪声减去无条件预测噪声'"**。

### 6.3 关键配对：ε_cond vs ε_uncond，差异是什么

> **ε_cond − ε_uncond 这个差，就是"文字条件让模型多识别出来的那部分噪声方向"。s 是把这个方向放大几倍——s 越大，文字越"用力"地拽生成结果。**

| 量 | 含义 | 例子（"jump high"） |
|---|---|---|
| ε_uncond | 无文字时模型预测的噪声方向 | 随便给一个动作的噪声估计——可能就是普通走路 |
| ε_cond | 给定 "jump high" 时的预测 | 噪声估计偏向"跳高"的动作方向 |
| **ε_cond − ε_uncond** | 文字带来的方向修正 | "比起普通走路，跳高需要往哪改" |
| s · (ε_cond − ε_uncond) | 修正放大 s 倍 | s=1 修正一倍，s=7 修正七倍 |

### 6.4 hand-calc 锚点：s 在 4 个档位的具体数值效果

设某一帧某一维上 ε_uncond = 0.10, ε_cond = 0.30。修正方向 ε_cond − ε_uncond = +0.20。

| s | 公式 | ε̃ | 生成倾向 |
|---|---|---|---|
| **0** | ε̃ = ε_uncond | 0.10 | **完全无视文字**——纯无条件采样 |
| **1** | ε̃ = ε_cond | 0.30 | 等于条件预测——文字"自然"贴合 |
| **2.5** | ε̃ = 0.10 + 2.5 × 0.20 = 0.60 | 0.60 | 文字方向放大 2.5 倍——MDM 默认 |
| **7** | ε̃ = 0.10 + 7 × 0.20 = 1.50 | 1.50 | 强力贴文字——MLD 默认 |

**注意一件反直觉的事**：s=1 时 ε̃ = ε_cond，**不是没用 CFG**——而是"自然条件预测"。CFG 只在 s > 1 时才真正"放大"。s = 0 反而是退化到无条件。

### 6.5 调参直觉表

| 用户反馈症状 | s 该调 | 同时观察 |
|---|---|---|
| **动作不像文字描述**（用户 A） | **调大 s**（默认 2.5 → 试 5 或 7.5） | 是否引入抖动 / 重复关键帧 |
| **动作很僵 / 同质化**（用户 B） | **调小 s**（默认 7.5 → 试 2.5 或 3.5） | 文字贴合是否掉太多 |
| 生成结果一致性差（多次跑差异大） | 调大 s（同上）+ fix random seed | FID 是否退化 |
| 生成结果一致性过强（同 prompt 出几乎一样） | 调小 s 增大多样性 | R-Precision 是否掉 |

### 6.6 为什么 MDM 和 MLD 的默认 s 不一样

MDM s=2.5，MLD s=7.5——为什么？

**直觉解释**：

- **MDM 在 raw motion 上跑扩散**，每一步更新的张量已经有几何 loss 拉着，文字方向不需要太用力。s 大了反而和几何约束打架。
- **MLD 在 latent 上跑扩散**，几何被锁在 VAE 里，扩散步本身没有几何信号，只能靠 CFG 强力拽住"语义"。s 设小了文本贴合度直接掉。

这给出一条 **TA 经验法则**：**raw 扩散 (MDM 系) s ∈ [2, 4]，latent 扩散 (MLD 系) s ∈ [5, 8]**。

### 6.7 训练时的 condition drop

CFG 推理离不开训练时的 condition drop——必须让模型同时学过"有条件"和"无条件"两种预测。具体做法：

```python
# 训练循环
if random() < 0.10:                    # MDM/MLD 都用 p_drop = 0.10
    cond = null_token                  # 用全零或特殊 token 替换 CLIP embedding
loss = mse(noise_pred, noise_target)
```

10% 这个数是 [Ho & Salimans 2021][S4] 的经验值，几乎所有 CFG 工作都用 0.10-0.15。设太小（< 5%）→ ε_uncond 学得不够，CFG 失灵；设太大（> 30%）→ ε_cond 学得不够，文字贴合差。

![fig9](/images/MDM-MLD-MotionDiffusion-DeepDive/fig9.png)

<details>
<summary>6.8 自检：训练时如果完全不做 condition drop，推理时还能用 CFG 吗？</summary>

不能。模型从未见过 "condition = ∅" 的输入，ε_uncond 的预测会变成乱来的随机方向，CFG 公式里 (ε_cond - ε_uncond) 这个差就成了噪声，s 放大也只是放大噪声。所以 condition drop 是 CFG 能用的**前提条件**——少这一行训练代码，整套 CFG 都失效。
</details>

## §7 选型决策与下游应用 [L1 Extended]

### 7.1 周一你交给老板的选型卡

到这里，MDM 和 MLD 的全部技术细节都在你手上了。老板要一句话决策："给我们的 NPC 动作生成 demo 用哪个？" 拿这个表去回答：

| 维度 | MDM | MLD | 选型建议 |
|---|---|---|---|
| **推理速度（A100, 196 帧）** | DDIM 50 步 ~0.6s | ~0.22s | 实时交互选 MLD |
| **质量 FID（HumanML3D）** | 0.544 | 0.473 | MLD 略好 |
| **R-Precision Top-3** | 0.611 | 0.772 | **MLD 明显好** |
| **几何精度（脚滑、关节漂）** | 几何 loss 直接控制，**精度高** | 受 VAE 重建质量限制 | 物理相关任务选 MDM |
| **训练复杂度** | 端到端 1 个 loss | 三阶段（VAE + Diffusion + 微调） | 资源紧张选 MDM |
| **下游可控性（精确控关节）** | 几何项可加新约束 | 需要重训 VAE | 需要精控选 MDM |
| **长序列（>10s）** | 可外推 | K=1 时容量受限，需加 K 重训 | 长动作选 MDM 或 K≥4 的 MLD |
| **UE 部署友好度** | ONNX 导出一个网络 | 要导 3 个网络（VAE-enc/Diff/VAE-dec） | 单网络选 MDM |

**简单决策树**：

- 客户端实时交互（< 100ms）+ 文本驱动 → **MLD**
- 离线批量生成 + 物理精度（地形交互、刚体） → **MDM**
- 想加几何约束扩展 → **MDM**
- 短动作（< 5s）+ 中等延迟可接受 → 都行，选你团队熟的

### 7.2 下游应用——MDM 系生态

MDM 的 x_0 预测 + 几何 loss 设计很容易"插新约束"，所以下游应用爆炸式生长：

- **PriorMDM**[S12]（ICLR 2024）：把训好的 MDM 当先验，做长序列拼接、双人动作配合（一人挥手另一人接）。核心 trick = 把多段 MDM 输出在 latent 上做加权融合。
- **OmniControl**[S12]（ICLR 2024）：把"任意关节子集的 3D 位置轨迹"当额外条件。例如"右手必须按这条曲线走"，模型生成的整段动作必须满足这个硬约束。
- **EDGE**[S12]（CVPR 2023）：把 MDM 思路搬到舞蹈，文本 condition 换成 Jukebox 提取的音乐特征。

### 7.3 UE 部署流程（MDM 路线）

如果你选 MDM 落到 UE，参考 4/26 PFNN 笔记 §8 的 ONNX 部署流程，差别在：

1. **MDM 网络要导出 8 层 Transformer**（PyTorch → ONNX → 用 ONNX Runtime C++ 在 UE 里加载）。
2. **每帧要做 50 次 forward**（DDIM 50 步），不能在 GameThread 里跑——必须放 Worker Thread（AsyncTask 或 FRunnable）。
3. **CLIP 文本编码可以预先在 Python 里算好**，把 512 维向量当 UE 端的 trigger 输入；或者干脆把 CLIP-ViT-B/32 也导成 ONNX。
4. **几何 loss 只在训练用，推理时不需要**——所以推理代码很干净。

**补充工程细节**：

- **UE 5.0+ Editor 可以用 Python**：通过 Python Editor Script Plugin，可以在 Editor 工具链里直接跑 PyTorch 推理（需要安装 PyTorch Python 包）。但 **Runtime（打包后）不支持 Python**，必须走 ONNX Runtime 或 TensorRT。
- **50 步在用户显卡上可能更慢**：论文报的 0.6s 是 A100；RTX 3060 可能要 2-3s，RTX 4090 约 0.3s。要做实时交互，必须上 **MotionLCM**（§8 预告）或用 **INT8 量化**（ONNX Runtime 支持）。
- **动画蓝图接入方式**：生成的是 196 帧 × 263 维序列，需要转成 UE 的 Animation Sequence 或 PoseAsset。推荐流程：MDM 生成 → 转 FBX（通过 Python 脚本）→ 导入 UE 成 Animation Sequence → 在 AnimGraph 里用。
- **MLD 路线在 UE 里更复杂**：需要导 3 个网络（VAE-enc + Latent Diffusion + VAE-dec），推理时要按顺序调用三次 ONNX Runtime，延迟累加。

### 7.4 现实主义提醒

- 论文报的 0.6s / 0.22s 是 A100；**你客户的笔记本 GPU 可能慢 5-10 倍**。要做实时交互，可能需要进一步用 MotionLCM（§8 预告）或 INT8 量化。
- HumanML3D 是用英文文本训练的；**中文 prompt 通常要先翻译**或重新对齐 CLIP。
- 论文的 demo 多在"日常动作"上漂亮（走、跑、跳）；**特殊动作（武术、舞蹈、特技）需要在自有数据集上 fine-tune**。

## §8 预告：MotionLCM 与一致性模型 [L1 Extended]

MLD 把扩散步从 1000 压到 50 已经很猛，但 50 步 × 5ms 仍是 250ms 量级，**真正的实时（< 30ms）还差一截**。2024 年的 [MotionLCM](https://arxiv.org/abs/2404.19759) 把 [Latent Consistency Model（LCM）](https://arxiv.org/abs/2310.04378) 思路搬到动作上，**把 50 步压到 1-4 步**，单段时间降到 ~30ms 量级。

核心思路：训练时用 consistency distillation，把"原扩散模型多步预测"蒸馏成"几步预测"，质量保持 90% 以上。这是 MLD 的下一代加速器，等到 Phase 2 / Phase 3 的笔记后续再单独深讲。

下一个值得追的方向：

- **Rectified Flow / Flow Matching 在动作上**（2024）：把扩散的曲折轨迹拉直，理论 1 步可达。
- **Diffusion Forcing**（NeurIPS 2024）：联合做"自回归生成 + 扩散去噪"，长序列质量大涨。
- **物理感知扩散**：在扩散步内插入 PD 控制器或刚体仿真，让生成的动作直接满足物理可执行性——这是 Phase 3 物理 RL 动画 与 motion diffusion 融合的方向。

> 📎 **衔接明天**：今天讲完了"扩散模型搬到动作上"这条线，明天（`2026-04-27`）我们换一个视角——**ASE（Adversarial Skill Embeddings）**，它把"风格控制"这件事从"文本条件化"变成了"潜空间 z 的方向"，和 MLD 的 Motion VAE 有异曲同工之处（都用 VAE 压缩动作），但 ASE 是 RL 路线而非扩散路线。如果你对"潜空间如何组织动作语义"有兴趣，ASE 是 DeepPhase（`2026-04-29`）的最佳前奏。

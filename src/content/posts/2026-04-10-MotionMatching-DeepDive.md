## 一、为什么你的怪物需要 Motion Matching？

### 1.1 传统方案的真实痛点

你的项目现在大概率在用这套方案：

```
┌─────────────────────────────────────────────────────┐
│              传统 Blend Space 方案                    │
│                                                     │
│  State Tree 决策 → 设置 Velocity / Direction        │
│       ↓                                             │
│  Anim BP 读取 → 驱动 Blend Space (8-Direction)      │
│       ↓                                             │
│  Blend Space 插值输出 → Final Pose                  │
│                                                     │
│  ❌ 问题清单：                                        │
│  · 方向突变时 = 数学插值过渡 = 滑步 / 旋转不自然       │
│  · 高低速切换 = 线性混合 = 脚部滑动明显               │
│  · 启动/停止 = 固定过渡时间 = 反应迟钝                │
│  · 地形适应 = 需要额外 IK 层 = 复杂度爆炸             │
└─────────────────────────────────────────────────────┘
```

**根本原因剖析：**

Blend Space 在本质上是一个**多维插值器**。它在你预设的几个关键点（Walk Forward, Walk Left, Run Forward 等）之间做数学插值。当角色需要的姿势不在这些关键点附近时，结果就是"猜出来的"中间帧——而真实的人/动物运动在过渡瞬间的姿态远比线性插值复杂。

具体来说，Blend Space 的三个根本限制是：
1. **离散化**：你只能放有限个动画到 Blend Space 中，覆盖不了所有速度×方向的组合
2. **线性插值假设**：两个姿势的"中间"不是简单的数学平均——真实运动有惯性、预判、重心转移
3. **无时序感知**：Blend Space 不知道角色"从哪个姿势过来"，只知道"当前要什么姿势"

### 1.2 Motion Matching 的核心承诺

```
┌─────────────────────────────────────────────────────┐
│              Motion Matching 方案                     │
│                                                     │
│  State Tree 决策 → 设置 Desired Velocity / Facing    │
│       ↓                                             │
│  FAnimNode_MotionMatching → 查询 PoseSearch Database │
│       ↓                                             │
│  找到最匹配的动捕帧 → 直接播放该帧附近的动画           │
│       ↓                                             │
│  输出 = **真实的动捕数据**，不是数学猜测               │
│                                                     │
│  ✅ 核心优势：                                        │
│  · 所有过渡都是真实录制过的动画帧                      │
│  · 启动/停止响应 < 100ms（取决于数据库覆盖）            │
│  · 转向自然度的质的飞跃                               │
│  · 高低速无缝切换（不需要显式的加速/减速状态）         │
│  · 一个节点解决 Locomotion，不需要复杂的状态机         │
└─────────────────────────────────────────────────────┘
```

### 1.3 一句话本质

> **MM 不是生成新动画，而是从一个巨大的"已录制动画帧数据库"中，找到最符合当前需求的那一帧，然后播放它。**

它就像一个超强的"动画搜索引擎"——你给它查询条件（我要往左前方跑、速度 300、当前左脚在前），它返回最佳匹配的动画帧位置。

### 1.4 MM vs Blend Space 深度对比

| 维度 | Blend Space | Motion Matching |
|------|-------------|-----------------|
| **数据源** | 手动放入的几个动画片段 | 大量动捕数据（分钟级别） |
| **中间帧** | 数学插值猜测 | 数据库中真实存在的帧 |
| **方向覆盖** | 通常 4~8 方向 | 数据库覆盖多少方向就有多少 |
| **速度覆盖** | 通常 2~3 个速度层级 | 连续速度范围的真实动画 |
| **过渡质量** | 依赖过渡时间设置 | 数据库中包含真实过渡帧 |
| **启动/停止** | 需要额外状态和过渡 | 数据库包含启停变体，自动选择 |
| **设置复杂度** | 低（拖动画进 Blend Space） | 中（配置 Schema + 构建数据库） |
| **运行时开销** | 极低 | 中等（KDTree 搜索 + 特征计算） |
| **内存占用** | 低 | 较高（数据库 + 索引） |
| **美术产出要求** | 少量精选动画 | 大量高质量动捕数据 |

**结论**：MM 不是 Blend Space 的"升级版"——它是一种完全不同的方法论。Blend Space 在数据少的场景下仍然有效（比如 NPC 背景角色），而 MM 适合需要高质量 Locomotion 的主角或重要怪物。

### 2.2 阶段 1：数据采集（动捕准备）

#### 怪物动捕的特殊要求

| 要求 | 人形角色 | 四足怪物 | 说明 |
|------|---------|---------|------|
| **移动方向数** | 8方向(N/NE/E/SE/S/SW/W/NW) | **16+ 方向** | 四足转向更复杂，前后腿协调方式不同 |
| **速度层级** | Walk/Jog/Sprint 3级 | **5+ 级** | 极慢爬行→慢走→正常跑→快跑→冲刺 |
| **启动/停止** | 各方向一组 | **各方向×各速度组合** | 不同速度下的启动惯性差异很大 |
| **转向幅度** | ±30°/±60°/±90° | **±15°~±180° 细粒度** | 四足原地转向和多步转向都需要 |
| **地形变体** | 平地为主 | **上坡/下坡/不平地面** | 四足的步态随地形变化更剧烈 |
| **总数据量** | ~5-10分钟 | **20-40 分钟** | 覆盖空间越大效果越好 |

### 2.3 阶段 2：特征提取（Feature Channel 完整架构）

> ⚠️ 这是整个 MM 流程中最核心的技术决策点！选错 Channel 或配错参数 = MM 效果直接拉垮。

#### 📐 [概念原理] 什么是 Feature Channel？

Feature Channel 是 PoseSearch 的核心抽象。每个 Channel 负责从每一帧动画中提取一个特定维度的数值特征。运行时，MM 用同样的 Channel 从当前角色状态构建查询向量，然后在数据库中搜索最相似的帧。

**整个过程就像搜索引擎：**
- Channel 定义 = 你在搜索什么（关键词类型）
- 数据库 Build = 给所有文档建索引
- 运行时查询 = 输入关键词搜索

#### 🔧 [引擎实现] Feature Channel 完整继承体系 [已验证]

**基类：`UPoseSearchFeatureChannel`** [已验证]
- 继承自 `UObject`, `IBoneReferenceSkeletonProvider`, `IPoseSearchFilter`
- 核心接口：
  - `Finalize(UPoseSearchSchema*)` — Schema 构建时调用
  - `BuildQuery(FSearchContext&)` — 运行时构建查询向量
  - `FillWeights(TArrayView<float>)` — 填充权重
  - `IndexAsset(FAssetIndexer&)` — 编辑器 Build 时索引动画数据
  - `IsFilterActive()` / `IsFilterValid()` — 姿势过滤

**复合通道（包含子通道的容器）：**

##### 1. `UPoseSearchFeatureChannel_Trajectory` [已验证] — 轨迹通道

这是 MM 最重要的通道之一。它不是一个原子特征，而是一个**复合通道**，内部根据你配置的 `FPoseSearchTrajectorySample` 数组自动展开为多个子通道。

```cpp
// 源码：PoseSearchFeatureChannel_Trajectory.h [已验证]

// 轨迹特征标志位（位掩码）
UENUM(meta = (Bitflags))
enum class EPoseSearchTrajectoryFlags : uint32
{
    Velocity            = 1 << 0,   // 3D 速度向量
    Position            = 1 << 1,   // 3D 位置偏移
    VelocityDirection   = 1 << 2,   // 归一化速度方向（3D）
    FacingDirection     = 1 << 3,   // 朝向方向（3D）
    VelocityXY          = 1 << 4,   // 水平面速度（2D，推荐！）
    PositionXY          = 1 << 5,   // 水平面位置（2D，推荐！）
    VelocityDirectionXY = 1 << 6,   // 水平面速度方向（2D）
    FacingDirectionXY   = 1 << 7,   // 水平面朝向（2D）
};

// 单个轨迹采样点
USTRUCT()
struct FPoseSearchTrajectorySample
{
    // 相对于当前采样时间的偏移（秒）
    // 负值 = 过去，正值 = 未来
    // 例如 -0.3 = 0.3秒前的轨迹，0.5 = 0.5秒后的预测轨迹
    float Offset = 0.f;

    // 位掩码：选择匹配哪些轨迹数据
    int32 Flags = int32(EPoseSearchTrajectoryFlags::Position);

    float Weight = 1.f;              // 该采样点的权重
    FName NormalizationGroup;        // 归一化分组
    FLinearColor DebugColor;         // 调试颜色
};

// Trajectory Channel 本体
UCLASS(meta = (DisplayName = "Trajectory Channel"))
class UPoseSearchFeatureChannel_Trajectory : public UPoseSearchFeatureChannel_GroupBase
{
    float Weight = 1.f;                        // 整体权重
    TArray<FPoseSearchTrajectorySample> Samples; // 采样点数组
    // 内部展开为子通道（Position/Velocity/Heading 等）
};
```

**🎯 怪物 Locomotion 的 Trajectory 推荐配置：**

```
Trajectory Channel 配置:
  整体 Weight = 1.0
  
  Samples 数组（7 个采样点）:
  ┌─────┬─────────┬──────────────────────────────────┬────────┐
  │ Idx │ Offset  │ Flags                            │ Weight │
  ├─────┼─────────┼──────────────────────────────────┼────────┤
  │  0  │ -0.4s   │ PositionXY                       │  0.6   │ ← 过去轨迹
  │  1  │ -0.2s   │ PositionXY | VelocityXY          │  0.8   │ ← 过去轨迹
  │  2  │  0.0s   │ VelocityXY | FacingDirectionXY   │  1.0   │ ← 当前
  │  3  │  0.1s   │ PositionXY | FacingDirectionXY   │  1.0   │ ← 近未来
  │  4  │  0.3s   │ PositionXY | FacingDirectionXY   │  0.9   │ ← 中未来
  │  5  │  0.5s   │ PositionXY                       │  0.7   │ ← 远未来
  │  6  │  0.8s   │ PositionXY                       │  0.4   │ ← 很远未来
  └─────┴─────────┴──────────────────────────────────┴────────┘
  
  为什么用 XY 版本？
  → 减少特征维度（2D 比 3D 少 1/3 维度）
  → 对于地面移动的怪物，水平面轨迹足够
  → 只有飞行/游泳的怪物才需要 3D 版本
  
  为什么过去和未来都要？
  → 过去轨迹：告诉 MM "我从哪里来"，帮助选择惯性一致的帧
  → 未来轨迹：告诉 MM "我要往哪里去"，帮助预判性选择
  → 两者结合 = 最自然的过渡
```

**⚠️ 四足怪物特殊调参：**

四足动物的骨盆（Root/Pelvis）在运动中有明显的左右摇摆和上下起伏。如果 Trajectory 使用根骨位置，会导致特征分布分散。

解决方案：
1. **使用 XY 版本** — 自动忽略垂直抖动
2. **降低 Trajectory 权重**，提高 Pose Channel 权重
3. **增加数据库密度** — 更多采样覆盖更多变体

##### 2. `UPoseSearchFeatureChannel_Pose` [已验证] — 姿态通道

另一个核心复合通道。负责匹配角色当前的骨骼姿态。

```cpp
// 源码：PoseSearchFeatureChannel_Pose.h [已验证]

// 骨骼特征标志位
UENUM(meta = (Bitflags))
enum class EPoseSearchBoneFlags : uint32
{
    Velocity  = 1 << 0,  // 关节速度
    Position  = 1 << 1,  // 关节位置
    Rotation  = 1 << 2,  // 关节旋转
    Phase     = 1 << 3,  // 运动相位（实验性）
};

// 单个骨骼配置
USTRUCT()
struct FPoseSearchBone
{
    FBoneReference Reference;   // 选择骨骼
    int32 Flags;                // 位掩码
    float Weight = 1.f;         // 该骨骼的权重
    FName NormalizationGroup;   // 归一化分组
};

// Pose Channel 本体
UCLASS(meta = (DisplayName = "Pose Channel"))
class UPoseSearchFeatureChannel_Pose : public UPoseSearchFeatureChannel_GroupBase
{
    float Weight = 1.f;
    TArray<FPoseSearchBone> SampledBones;   // 要采样的骨骼列表
    
    EInputQueryPose InputQueryPose = EInputQueryPose::UseContinuingPose;
    // ⚠️ 重要！默认 UseContinuingPose 表示优先复用继续姿势的数据
    // 而不是每帧都从角色实际姿态重新计算（性能优化）
    
    bool bUseCharacterSpaceVelocities = true;
    // true = 速度在角色空间计算（推荐，对旋转不敏感）
    // false = 速度在世界空间计算
};
```

  整体 Weight = 1.0
  InputQueryPose = UseContinuingPose
  bUseCharacterSpaceVelocities = true

  SampledBones 数组:

  │ Idx │ Bone         │ Flags                  │ Weight │

#### 📐 [概念原理] 特征维度如何影响性能和质量

每增加一个特征维度，就意味着：
- 数据库内存 **线性增长**
- PCA 降维的有效性 **递减**
- 搜索精度 **提高**但搜索时间 **增长**

**最佳实践：** 使用 XY 版本代替 3D 版本可以减少 1/3 维度，通常不影响地面 Locomotion 的质量。

```
特征维度计算示例:

Trajectory Channel (7 个采样点):
  · 采样点 0: PositionXY = 2 维
  · 采样点 1: PositionXY + VelocityXY = 4 维
  · 采样点 2: VelocityXY + FacingDirectionXY = 4 维
  · ...
  → 小计约 20 维

Pose Channel (7 个骨骼):
  · 每个骨骼 Position = 3 维, Velocity = 3 维 → 6 维
  · 7 × 6 = 42 维
  → 如果用 XY 版本: 7 × 4 = 28 维

总计: ~48-62 维 → Schema.SchemaCardinality

PCA 降到 4 维 (NumberOfPrincipalComponents=4) → KDTree 搜索在 4 维空间进行
```

#### 🔧 [引擎实现] Continuing Pose 机制详解 [已验证]

这是 MM 最重要的运行时机制。每帧搜索时，MM 会做两件事：
1. **全搜索** — 在整个数据库中找最佳匹配
2. **Continuing Pose Search** — 评估"继续播放当前动画"的 Cost

然后比较两者 Cost，选更低的。这样 MM 就不会每帧都跳到新动画，而是在当前动画"还不错"的时候继续播放。

```
Continuing Pose 的 Cost 计算:

ContinuingCost = 正常特征匹配 Cost 
               + ContinuingPoseCostBias (默认 -0.01，负值=更容易被选)
               + LoopingCostBias (如果是循环动画)

全搜索 BestCost = 在整个数据库中找到的最低 Cost

if ContinuingCost < BestCost:
    → 继续播放当前动画
else:
    → 跳转到全搜索的最佳帧，触发 Blend 过渡
```

**EPoseSearchInterruptMode** [已验证] — 控制 Continuing Pose 的中断策略：

```
DoNotInterrupt (默认)
  → 正常执行 Continuing Pose Search
  → 最平滑的行为

InterruptOnDatabaseChange
  → 如果当前动画的数据库不在搜索列表中 → 中断
  → 用于切换数据库时

InterruptOnDatabaseChangeAndInvalidateContinuingPose
  → 中断 + 失效继续姿势
  → 强制从 Pose History 重建查询

ForceInterrupt
  → 强制中断，不执行 Continuing Pose Search
  → 用于状态大转换（如从 Locomotion 切到 Combat）

ForceInterruptAndInvalidateContinuingPose
  → 强制中断 + 失效
  → 最激进的中断
```

#### 📐 [概念原理] PCAKDTree 搜索算法详解

MM 的默认搜索算法是 PCAKDTree，它分三步：

```
Step 1: PCA 降维
  原始特征向量 (例如 50 维)
    ↓
  PCA 投影到 4 维空间 (NumberOfPrincipalComponents=4)
    ↓
  低维向量 (4 维)

  PCA 的作用:
  · 去除冗余特征（比如左脚和右脚的运动往往相关）
  · 保留最重要的变化方向
  · 大幅降低搜索空间维度

Step 2: KDTree KNN 搜索
  在 4 维 PCA 空间的 KDTree 中搜索
    ↓
  返回前 K 个最近邻 (KDTreeQueryNumNeighbors=200)
    ↓
  这 200 个候选帧是 PCA 空间的"近似最优"

Step 3: 精确 Cost 评估
  对 200 个候选帧用原始 50 维特征计算精确 Cost
    ↓
  选出 Cost 最低的帧
    ↓
  考虑各种偏置（Continuing, Looping, Base）
    ↓
  返回最终结果

⚠️ 特殊优化: 
  如果 KDTreeQueryNumNeighbors = 1
  → 不存储原始 Values 数组！
  → 完全依赖 PCA 空间的近似搜索
  → 大幅节省内存，但精度下降
```

## 四、MM + State Machine 分层架构（实战重点！）

### 4.1 为什么必须分层？

| MM 的强项 ✅ | MM 的弱项 ❌ |
|---|---|
| 连续的、参数化控制的行为（Locomotion） | 离散的状态切换（Attack/Death） |
| 平滑的方向/速度变化 | 需要精确 Notify 时机的动画（VFX 触发） |
| 快速响应的启动/停止 | 有明确开始和结束的一次性行为 |
| 不需要定义状态和过渡 | 需要严格的状态流程控制 |

**State Machine 的强项 = MM 的弱项！所以分层使用，各自发挥优势。**

在 Skeleton Editor 中创建 Blend Profile:

  ☑ Spine_01        Weight = 1.0
  ☑ Spine_02        Weight = 1.0
  ☑ Spine_03        Weight = 1.0 (如果有)

### 4.4 State Machine 上半身状态设计

```
UpperBody State Machine:

┌──────────────┐
│    Entry     │

       │ (Default)
       ▼
┌──────────────┐     ┌──────────────┐
│ Idle_Upper   │◄────│   Roar       │

│·耳朵转动     │     └──────┬───────┘
└──────┬───────┘            │ (结束后回 Idle)
       │ (进入Combat)
       ▼
┌──────────────┐     ┌──────────────┐
│ CombatReady  │◄────│  Stun/Hurt   │
│ (战斗待机)    │     │  (受伤/眩晕)  │
│·低姿态警戒   │     └──────┬───────┘
└───┬────┬────┘            │ (死亡)
    │    │                 ▼
    │    │          ┌──────────────┐
    │    └──────────►│    Death     │
    │   (死亡)       │  (死亡)      │
    ▼               └──────────────┘
┌─────────────────┐
│   Attack_State  │
│ (攻击选择器)     │

过渡类型:
  Idle → CombatReady:  Inertialization (0.2s)
  CombatReady → Attack: Standard (0.1s) 
  Attack → CombatReady: Inertialization (0.15s)
  Any → Death:         Inertialization (0.1s)
```

### 4.5 两个系统之间的通信机制

```
Event Graph 每帧 Tick:
  ┌──────────────────────────────────┐
  │ 1. 从 State Tree 读取 Intent     │
  │                                  │
  │ 2. 根据 Intent 影响 Movement:    │
  │    → 直接设置 CharacterMovement  │
  │      的 DesiredVelocity          │
  │    → MM 通过 Trajectory 自动感知 │
  │    → 不需要手动传参给 MM 节点！   │
  │                                  │
  │ 3. 根据 Intent 切换 SM:          │
  │    → 设置 UpperBodyState 变量    │
  │    → SM 根据变量自动过渡          │
  │                                  │
  │ 4. 特殊情况:                      │
  │    · 攻击: 调用 SetInterruptMode │
  │      (ForceInterrupt) 让 MM      │
  │      立即重新搜索攻击时的站立姿态  │
  │    · 眩晕: SetDatabaseToSearch   │
  │      (DB_StunLocomotion) 切换到  │
  │      眩晕专用移动库               │
  │    · 死亡: bShouldSearch=false   │
  │      停止搜索，冻结最后姿态       │
  └──────────────────────────────────┘
```

### 5.2 常见问题诊断完全手册

| 症状 | 可能原因 | 诊断方法 | 解决方案 |
|------|---------|---------|---------|
| **角色滑步严重** | Pose Channel 缺少脚部 Velocity | Debugger 查看 Pose 特征是否包含脚部速度 | 添加脚部骨骼的 Velocity Flag，提高权重到 1.2+ |
| **转向时抽搐/抖动** | Trajectory 未来偏移太短 | 3D 可视化看 Trajectory 预测是否太短 | 增加 Samples 中 Offset > 0.3s 的采样点 |
| **转向太迟钝** | Trajectory 权重太低 / 未来采样点太少 | Debugger 看 Trajectory Cost 在总 Cost 中的占比 | 提高 Trajectory Weight 或增加未来采样点 |
| **启动/停止不自然** | 数据库缺少启停变体 | 统计数据库中 start/stop 帧数 | 录制更多 start/stop 动画加入数据库 |
| **突然跳帧** | ContinuingPoseCostBias 太接近 0 | Debugger 比较 Continuing Cost vs Best Cost | 调低到 -0.02 ~ -0.05 |
| **长时间不换动画** | ContinuingPoseCostBias 太负 | Debugger 看 Continuing Cost 是否总是最低 | 调回 -0.01 ~ -0.02 |
| **高速时动画乱套** | 数据库缺少高速动画 | 查看数据库中最高速度的动画 | 录制冲刺速度的动捕加入 |
| **两只脚交叉** | 缺少 Phase 信息 | 观察跳转瞬间的脚步相位 | 添加 Phase Channel [实验性] 或 FilterCrashingLegs Channel |
| **内存占用过大** | 数据库帧数太多 / PCA 维度不够 | 查看 Database Statistics | 增大 PosePruningSimilarityThreshold / 减少 Channel |
| **CPU 占用高** | 每帧搜索 + 怪物太多 | Profiler 看 PoseSearch 时间 | 增大 SearchThrottleTime / 远处角色回退 BS |
| **同类怪物动作完全同步** | 共享 Database 且初始状态相同 | 观察多个怪物 | 随机化初始动画时间 / 使用 Permutation 系统 |

### 5.3 性能优化 Checklist

```
□ Schema 优化 (影响数据库构建和内存):
  · ✅ 用 XY 变体代替 3D → 减少 1/3 维度
  · ✅ SampleRate = 30（不超过动捕帧率）
  · ✅ bAddDataPadding = true → SIMD 加速
  · ✅ 只采样关键骨骼（不是全部骨骼）
  · ⚠️ 移除不必要的 Channel

□ Database 优化 (影响内存和搜索速度):
  · ✅ PoseSearchMode = PCAKDTree（默认最优）
  · ✅ NumberOfPrincipalComponents = 4~8
  · ✅ PosePruningSimilarityThreshold > 0（去重节省内存）
  · ⚠️ KDTreeQueryNumNeighbors: 生产环境可降到 100
  · ⚠️ 如果内存紧张: KDTreeQueryNumNeighbors = 1（牺牲精度换内存）

□ 运行时优化 (影响帧率):
  · ✅ SearchThrottleTime = 0.03~0.05（不必每帧搜索）
  · ✅ 静止时 bShouldSearch = false
  · ✅ 场景中同类怪物共享同一个 Database
  · ✅ 远处角色回退到 Blend Space（距离 LOD）
  · ⚠️ 考虑怪物分组错开搜索帧

□ 内存占用参考:
  · 10 分钟动捕 + 50 维 Schema ≈ 3~8 MB
  · 30 分钟动捕 + 50 维 Schema ≈ 10~25 MB
  · (取决于 PCA 设置和剪枝)
```

## 六、进阶技巧

### 6.1 多数据库切换策略

```cpp
// 在 Event Graph 中根据状态切换数据库 [已验证 API]

// 场景 1: 进入战斗 → 切换到战斗 Locomotion 库
MotionMatchingNode.SetDatabaseToSearch(
    DB_CombatLocomotion,
    EPoseSearchInterruptMode::InterruptOnDatabaseChange
);

// 场景 2: 受伤 → 切换到跛行库
MotionMatchingNode.SetDatabaseToSearch(
    DB_InjuredLocomotion,
    EPoseSearchInterruptMode::ForceInterrupt
);

// 场景 3: 恢复正常 → 重置为默认库
MotionMatchingNode.ResetDatabasesToSearch(
    EPoseSearchInterruptMode::InterruptOnDatabaseChange
);

// 场景 4: 多库同时搜索（选最优）
TArray<UPoseSearchDatabase*> Databases = {DB_Walk, DB_Run, DB_Sprint};
MotionMatchingNode.SetDatabasesToSearch(
    Databases,
    EPoseSearchInterruptMode::DoNotInterrupt
);
```

### 6.2 🔧 [引擎实现] NormalizationSet — 跨库统一归一化 [已验证]

当你有多个数据库（Idle库 vs 跑步库），不同库的特征值范围差异很大。`UPoseSearchNormalizationSet` [已验证] 允许统一归一化：

```
场景: 
  DB_Idle (速度 0~50) → 归一化后 Velocity 特征 ≈ 0~0.1
  DB_Run  (速度 200~600) → 归一化后 Velocity 特征 ≈ 0.4~1.0
  
  如果分别归一化 → Cost 不可比！Idle 的 Cost 永远很低
  
  用 NormalizationSet 把两个库统一归一化:
  → Velocity 在 0~600 范围内归一化
  → Cost 变得可以跨库比较
```

### 6.3 🔧 [引擎实现] Interaction 系统 [实验性]

源码中有完整的多角色交互 MM 系统：
- `FAnimNode_MotionMatchingInteraction` [已验证]
- `UPoseSearchInteractionAsset` [已验证]
- `UPoseSearchInteractionSubsystem` [已验证]

这允许两个角色的动画同步匹配（如格斗、抓取）。⚠️ 实验性，生产慎用。

### 6.4 Mirror 支持

Schema 中每个 Skeleton 可以配置 `UMirrorDataTable`。配置后，每个动画资产可以设置 `EPoseSearchMirrorOption` [已验证]：
- `UnmirroredOnly` — 只用原始版本
- `MirroredOnly` — 只用镜像版本
- `UnmirroredAndMirrored` — 两个版本都入库

这意味着你可以用一半的动画覆盖两个方向！

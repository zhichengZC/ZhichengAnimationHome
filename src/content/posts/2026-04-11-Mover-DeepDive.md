## 第一章：Mover 插件概述与设计哲学

### 1.1 为什么需要 Mover？

UE 传统的角色移动依赖 `UCharacterMovementComponent`（以下简称 CMC），这是一个诞生于 UE4 早期的「巨石类」（Monolithic Class）。CMC 将行走、跳跃、游泳、飞行等所有运动逻辑封装在一个超过 10000 行的单一组件中，存在以下问题：

1. **扩展困难**：添加新运动模式（如攀爬、滑翔、钩锁）需要深入修改 CMC 内部逻辑
2. **代码耦合**：物理检测、运动模式切换、网络同步、输入处理全部耦合在一起
3. **调试困难**：难以隔离特定运动模式的 Bug
4. **AI 集成受限**：AI 角色和玩家角色共用同一套输入管线，但需求差异大
5. **异步模拟受限**：CMC 的紧耦合设计难以支持物理线程异步模拟

Mover 插件的设计目标是用 **组合式（Compositional）** 和 **声明式（Declarative）** 的方式重新定义角色运动系统。

### 1.2 声明式 vs 命令式运动

| 维度 | CMC（命令式） | Mover（声明式） |
|------|-------------|----------------|
| **运动模式定义** | 硬编码枚举 `EMovementMode` | `FName` 注册，任意扩展 |
| **模式切换** | `SetMovementMode()` 直接设置 | `QueueNextMode()` 排队，下一帧生效 |
| **速度控制** | 直接修改 `Velocity` 属性 | 通过 Layered Move / Mode 的 `GenerateMove` 声明意图 |
| **网络同步** | 自定义的 Move/Ack/Correction 流程 | Network Prediction Plugin 或 Chaos Physics 统一管理 |
| **扩展方式** | 继承重写虚函数 | 组合 Mode + Transition + LayeredMove 对象 |
| **输入处理** | 直接读取 `GetCharacterMovement()->XXX` | 通过 `IMoverInputProducerInterface` 声明式提交 InputCmd |

| `OnMovementModeChanged` | 模式切换时 | ✅ | 播放动画、VFX |
| `OnTeleportSucceeded` | 传送成功 | ✅ | UI 提示等 |
| `OnTransitionTriggered` | 过渡规则触发 | ✅ | 调试、日志 |
| `OnPostFinalize` | 帧定稿后（含重模拟） | ✅ 保证游戏线程 | 最终状态同步 |
| `ProcessGeneratedMovement` | 提议移动生成后 | - | 最终修改提议移动 |

### 2.4 三阶段 Tick 系统

Mover 的核心运行循环分三个阶段，定义在 `EMoverTickPhase` 枚举中：

```cpp
// 源码验证：MoverTypes.h:27-39
UENUM(BlueprintType)
enum class EMoverTickPhase : uint8
{
    Invalid = 0,
    ProduceInput = 1,       // 阶段一：生产输入
    SimulateMovement = 2,   // 阶段二：模拟移动
    ApplyState = 3          // 阶段三：应用状态
};
```

**阶段一：ProduceInput**

```cpp
// 源码验证：MoverComponent.cpp:340-353
void UMoverComponent::ProduceInput(const int32 DeltaTimeMS, FMoverInputCmdContext* Cmd)
{
    Cmd->InputCollection.Empty();
    for (TObjectPtr<UObject> InputProducerComponent : InputProducers)
    {
        if (IsValid(InputProducerComponent))
        {
            IMoverInputProducerInterface::Execute_ProduceInput(
                InputProducerComponent, DeltaTimeMS, IN OUT *Cmd);
        }
    }
    CachedLastProducedInputCmd = *Cmd;
}
```

关键点：
- 遍历所有实现 `IMoverInputProducerInterface` 的组件
- 每个组件可以往 `InputCollection` 中添加自己的输入数据
- 如果 `bGatherInputFromAllInputProducerComponents` 为 true，会自动收集 Owner 上所有实现该接口的组件

**阶段二：SimulateMovement**

```cpp
// 源码验证：MoverComponent.cpp:527-684
void UMoverComponent::SimulationTick(const FMoverTimeStep& InTimeStep,
    const FMoverTickStartData& SimInput, OUT FMoverTickEndData& SimOutput)
{
    // 1. 处理回滚
    if (bHasRolledBack) { ProcessFirstSimTickAfterRollback(InTimeStep); }
    
    // 2. Pre-Simulation 回调
    PreSimulationTick(MoverTimeStep, SimInput.InputCmd);
    
    // 3. 检查外部移动
    CheckForExternalMovement(SimInput);
    
    // 4. 持久化 Sync State 数据
    for (const FMoverDataPersistence& Entry : PersistentSyncStateDataTypes) { ... }
    
    // 5. 处理 SuggestedMovementMode 输入
    if (Input && !Input->SuggestedMovementMode.IsNone())
        ModeFSM->QueueNextMode(Input->SuggestedMovementMode);
    
    // 6. 核心：状态机 Tick（在这里执行实际的移动计算）
    ModeFSM->OnSimulationTick(UpdatedComponent, UpdatedCompAsPrimitive,
        SimBlackboard.Get(), SimInput, MoverTimeStep, SimOutput);
    
    // 7. Post-Movement 回调
    OnPostMovement.Broadcast(MoverTimeStep, SimOutput.SyncState, SimOutput.AuxState);
    
    // 8. 基于移动的调度更新
    if (bSupportsKinematicBasedMovement)
        UpdateBasedMovementScheduling(SimOutput);
    
    // 9. Post-Simulation 回调
    OnPostSimulationTick.Broadcast(MoverTimeStep);
}
```

**阶段三：ApplyState**

由 Backend Liaison 在适当时机调用 `FinalizeFrame()`，将模拟结果应用到实际组件上。

## 第五章：Layered Move 叠加运动系统

### 5.1 概念

Layered Move 是**临时的附加运动**，类似 CMC 的 Root Motion Source。它们只生成提议移动，依赖当前 Movement Mode 来执行。

### 5.2 与 CMC Root Motion Source 对比

| 特性 | CMC Root Motion Source | Mover Layered Move |
|------|----------------------|-------------------|
| 生命周期 | Duration + Strength | Duration + 自定义 |
| 混合 | 累加 | 通过 MovementMixer 混合 |
| 网络同步 | 手动序列化 | 自动包含在 SyncState 中 |
| 跨模式存活 | 不可以 | 可以（可配置） |
| 蓝图支持 | 有限 | 完整蓝图支持 |

### 5.3 内置 Layered Move 类型

| 类型 | 文件 | 说明 |
|------|------|------|
| `FLayeredMove_LinearVelocity` | BasicLayeredMoves.h | 恒定线速度 |
| `FLayeredMove_DirectionalForce` | BasicLayeredMoves.h | 方向力 |
| `FLayeredMove_RadialForce` | BasicLayeredMoves.h | 径向力 |
| `FLayeredMove_HomingForce` | BasicLayeredMoves.h | 追踪力 |
| `FLayeredMove_Launch` | LaunchMove.h | 发射（类似 CMC LaunchCharacter） |
| `FLayeredMove_MultiJump` | MultiJumpLayeredMove.h | 多段跳 |
| `FLayeredMove_AnimRootMotion` | AnimRootMotionLayeredMove.h | 动画 Root Motion |
| `FLayeredMove_RootMotionAttribute` | RootMotionAttributeLayeredMove.h | Root Motion 属性 |

## 第六章：Movement Modifier 运动修改器

### 6.1 概念

Movement Modifier 间接影响运动模拟，而不直接提议或执行移动。常见用途是蹲伏（Stance）等需要多个设置同时变化的机制。

### 6.2 内置 Modifier

| 类型 | 说明 |
|------|------|
| `FStanceModifier` | 蹲伏/站立姿态切换 |
| `FPhysicsStanceModifier` | 物理驱动版本的姿态切换 |

### 6.3 生命周期管理

## 第二十二章：网络同步深入

### 22.1 与 CMC 网络模型的区别

**CMC 模型**：客户端发送 Input+State 的 "Move" → 服务器执行 → 比较 → 发送 Ack 或 Correction

**Mover / NPP 模型**：
- 所有端点在共享时间线上模拟
- 客户端预测性地领先服务器少量时间
- 客户端只发送 Input（不发 State）
- 服务器缓冲输入直到对应帧
- 服务器广播状态 → 客户端自行决定是否回滚+重模拟

### 22.2 回滚机制

```cpp
// 源码验证：MoverComponent.cpp:355-377
void UMoverComponent::RestoreFrame(
    const FMoverSyncState* SyncState,
    const FMoverAuxStateContext* AuxState,
    const FMoverTimeStep& NewBaseTimeStep)
{
    // 1. 通知 Pre-Rollback（旧的无效状态 + 新的有效状态）
    OnSimulationPreRollback(&InvalidSyncState, SyncState, ...);
    // 2. 恢复组件状态
    SetFrameStateFromContext(SyncState, AuxState, /* rebase? */ true);
    // 3. 通知 Post-Rollback
    OnSimulationRollback(SyncState, AuxState, NewBaseTimeStep);
}
```

## 第二十四章：性能考虑与最佳实践

### 24.1 性能优化建议

1. **使用 NavWalkingMode**：AI 怪物优先使用 NavMesh 行走，跳过碰撞 Sweep
2. **Grouped Movement**：大量 NPC 启用 `bUseDeferredGroupMovement` 提升性能
3. **固定 Tick + 平滑**：设置 Fixed Tick 模拟 + `VisualComponentOffset` 平滑
4. **Async Modes**：对支持异步的模式使用异步模拟
5. **Sim Blackboard 缓存**：利用黑板缓存计算结果避免重复计算

### 24.2 常见陷阱

1. 不要在模拟外直接修改 Actor Transform
2. 不要依赖 Sim Blackboard 在回滚后仍有效
3. `QueueNextMode()` 是排队而非立即生效
4. Cooked 构建可能丢失数据，禁用 `bOptimizeBPComponentData`
5. 同一类型只能有一个活跃 Modifier

### 25.1 核心收获

Mover 插件代表了 UE5 角色移动系统的未来方向：
- **模块化**：Mode + Transition + LayeredMove + Modifier 的组合式设计

3. Layered Move 非常适合怪物的攻击位移、被击效果

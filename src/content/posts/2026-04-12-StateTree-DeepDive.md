## 第一章：State Tree 概述与设计哲学

### 1.1 State Tree 是什么？

State Tree 是 UE5 引入的**通用分层状态机**（General Purpose Hierarchical State Machine）。它的定位不是替代 Behavior Tree，而是提供一种更**轻量、可视化、组合式**的行为描述系统。

**核心设计理念：**

1. **数据驱动**：State Tree 是 `UDataAsset`，不是蓝图类，意味着它是纯数据资产，可以热重载
2. **Schema 约束**：每个 State Tree 都绑定一个 Schema，Schema 决定了这棵树可以使用哪些 Task、Condition、Evaluator
3. **组合式架构**：状态（State）+ 任务（Task）+ 条件（Condition）+ 转换（Transition）四元组，通过组合而非继承来构建行为
4. **属性绑定**：强大的 Property Binding 系统，Task 之间、Task 与外部数据之间都可以通过绑定传递数据
5. **分层执行**：支持 Linked State、Linked Asset 和 Subtree，可以将复杂行为分层拆解

### 1.2 State Tree vs Behavior Tree

| 维度 | Behavior Tree | State Tree |
|------|---------------|------------|
| **设计模式** | 树形决策（Selector/Sequence/Decorator） | 分层状态机（State/Transition/Task） |
| **执行模型** | 每帧从根重新遍历 | 维持当前状态，仅在转换触发时切换 |
| **数据传递** | Blackboard（全局键值对） | Property Binding（编译期绑定） |
| **扩展方式** | BTTask/BTService/BTDecorator（UObject） | FStateTreeTaskBase（FStruct，更轻量） |
| **内存开销** | UObject 实例 × 节点数 | Struct 实例，紧凑内存布局 |
| **调试工具** | Behavior Tree Debugger | State Tree Debugger + Rewind Debugger |
| **Schema 约束** | 无 | 有，Schema 控制可用节点类型 |
| **并行执行** | Parallel Composite | Run Parallel Tree Task |
| **适用场景** | 复杂 AI 决策树 | 状态驱动的行为（怪物状态机、交互系统） |

### 1.3 你项目中的应用

在你的天美肉鸽项目中，怪物使用 **State Tree + Anim State Machine** 架构：
- State Tree 负责**行为决策层**：巡逻、追击、攻击、受击、死亡等状态切换
- Anim State Machine 负责**动画表现层**：具体的动画播放、混合、过渡

State Tree 通过 `UStateTreeComponent` 挂在怪物 Actor 上，每帧 Tick 执行状态逻辑和转换评估。

### 3.4 FStateTreeExecutionContext 的 Tick 流程

这是最关键的——每帧 `UStateTreeComponent::TickComponent` 会调用 `FStateTreeExecutionContext::Tick(DeltaTime)`，内部流程：

```
Tick(DeltaTime)
├── TickUpdateTasks(DeltaTime)
│   ├── 复制绑定属性（Property Copy）
│   ├── Tick Global Evaluators
│   ├── Tick Global Tasks
│   └── Tick Active State Tasks
│       └── 对每个活跃状态的每个 Task 调用 Task::Tick()
│           └── 如果 Task 返回 Succeeded/Failed → 标记状态完成
└── TickTriggerTransitions()
    ├── 对每个活跃状态（从叶到根）：
    │   ├── Task::TriggerTransitions()（如果 bShouldAffectTransitions）
    │   ├── 检查 OnStateCompleted 转换
    │   ├── 检查 OnTick 转换
    │   └── 检查 OnEvent 转换
    ├── 找到最高优先级的可执行转换
    ├── 退出旧状态（ExitState）
    ├── 选择新状态（State Selection）
    └── 进入新状态（EnterState）
```

### 4.2 状态类型（EStateTreeStateType）

```cpp
enum class EStateTreeStateType : uint8
{
    State,       // 包含 Task 和子状态的普通状态
    Group,       // 只包含子状态（无 Task），用于组织结构
    Linked,      // 链接到树中另一个状态（执行跳转到目标状态）
    LinkedAsset, // 链接到另一个 StateTree 资产
    Subtree,     // 可被 Linked 引用的子树
};
```

**📐 概念原理**：
- `State` 是最常用的，你项目中的 Idle、Chase、Attack 等都是 State 类型
- `Group` 用于逻辑分组，比如"战斗状态组"下面包含 Attack/Dodge/Block 子状态
- `Linked` 相当于状态机里的"跳转"，可以复用同一棵树里的子树
- `LinkedAsset` 更强大，可以引用另一个 StateTree 资产，实现模块化

### 4.3 子状态选择行为（EStateTreeStateSelectionBehavior）

```cpp
enum class EStateTreeStateSelectionBehavior : uint8
{
    None,                                          // 不可直接选中
    TryEnterState,                                 // 直接进入（不检查子状态）
    TrySelectChildrenInOrder,                      // 按顺序尝试子状态
    TrySelectChildrenAtRandom,                     // 随机选择子状态
    TrySelectChildrenWithHighestUtility,           // 选择 Utility 最高的子状态
    TrySelectChildrenAtRandomWeightedByUtility,    // 按 Utility 权重随机
    TryFollowTransitions,                          // 按转换规则走
};
```

**实战建议**：
- 怪物的行为选择：用 `TrySelectChildrenWithHighestUtility` + Consideration 系统来做基于评分的决策（类似 Utility AI）
- 简单的优先级状态机：用 `TrySelectChildrenInOrder`，第一个满足条件的子状态被选中

### 5.2 转换触发类型（EStateTreeTransitionTrigger）

```cpp
enum class EStateTreeTransitionTrigger : uint8
{
    None           = 0,
    OnStateCompleted = 0x1 | 0x2,  // 状态完成时（成功或失败）
    OnStateSucceeded = 0x1,         // 状态成功时
    OnStateFailed    = 0x2,         // 状态失败时
    OnTick           = 0x4,         // 每帧检查
    OnEvent          = 0x8,         // 收到特定事件时
};
```

### 5.3 转换目标类型（EStateTreeTransitionType）

```cpp
enum class EStateTreeTransitionType : uint8
{
    None,              // 无转换
    Succeeded,         // 停止树/子树，标记成功
    Failed,            // 停止树/子树，标记失败
    GotoState,         // 跳转到指定状态
    NextState,         // 跳转到下一个兄弟状态
    NextSelectableState, // 跳转到下一个可选择的兄弟状态
};
```

### 5.4 转换优先级与评估顺序

转换评估的顺序是**从叶状态到根状态**（leaf to root）。在同一帧中，多个转换可能同时满足条件，此时按以下规则决定哪个生效：

1. **第一个被访问的**（即叶状态优先）
2. **优先级最高的**（`EStateTreeTransitionPriority`）
3. **能成功完成状态选择的**

**实战示例**：怪物在 Attack 状态，同时：
- Attack 的 `OnStateCompleted → Idle` 转换就绪
- 父状态 Combat 的 `OnEvent("TakeDamage") → Hit` 转换就绪

由于 Attack 是叶状态，它的转换先被检查。但如果 Hit 转换的 Priority 设为 `High`，则 Hit 优先。

## 第六章：任务（Task）系统

### 6.1 FStateTreeTaskBase — 任务基类

**源码路径**：`Plugins/Runtime/StateTree/Source/StateTreeModule/Public/StateTreeTaskBase.h`
**基类**：`FStateTreeNodeBase`

Task 是 State Tree 中执行实际逻辑的结构体。注意它是 `USTRUCT` 而不是 `UCLASS`，这是为了更紧凑的内存布局。

**🔧 虚函数（引擎实现）：**

```cpp
// 进入状态（返回 Running 继续，Succeeded/Failed 立即完成）
virtual EStateTreeRunStatus EnterState(
    FStateTreeExecutionContext& Context, 
    const FStateTreeTransitionResult& Transition) const;

// 退出状态
virtual void ExitState(
    FStateTreeExecutionContext& Context, 
    const FStateTreeTransitionResult& Transition) const;

// 每帧 Tick（仅 bShouldCallTick 为 true 时调用）
virtual EStateTreeRunStatus Tick(
    FStateTreeExecutionContext& Context, 
    const float DeltaTime) const;

// 状态完成回调（反向顺序调用，用于传播状态给更早的 Task）
virtual void StateCompleted(
    FStateTreeExecutionContext& Context, 
    const EStateTreeRunStatus CompletionStatus, 
    const FStateTreeActiveStates& CompletedActiveStates) const;

// 触发转换（仅 bShouldAffectTransitions 为 true 时调用）
virtual void TriggerTransitions(FStateTreeExecutionContext& Context) const;
```

### 6.2 Task 标志位

```cpp
uint8 bShouldStateChangeOnReselect : 1;  // 重选时是否触发状态变化（默认 true）
uint8 bShouldCallTick : 1;                // 是否每帧 Tick（默认 true）
uint8 bShouldCallTickOnlyOnEvents : 1;    // 仅有事件时 Tick（默认 false）
uint8 bShouldCopyBoundPropertiesOnTick : 1; // Tick 前是否复制绑定属性（默认 true）
uint8 bShouldCopyBoundPropertiesOnExitState : 1; // ExitState 前是否复制（默认 true）
uint8 bShouldAffectTransitions : 1;        // 是否影响转换（默认 false）
uint8 bConsideredForScheduling : 1;        // 是否参与调度（默认 true）
uint8 bTaskEnabled : 1;                    // 是否启用（默认 true）
```

**📐 概念原理 — bShouldStateChangeOnReselect**：
- `true`（默认）：适用于**动作型 Task**，如播放动画。即使状态被重选，也要重新执行 EnterState/ExitState
- `false`：适用于**状态型 Task**，如占用资源。状态被重选时保持不变，避免重复获取/释放

### 6.4 自定义 Task 开发

**方式一：C++ Struct（推荐）**

```cpp
// 实例数据
USTRUCT()
struct FMyTaskInstanceData
{
    GENERATED_BODY()
    
    UPROPERTY(EditAnywhere, Category = "Input")
    TObjectPtr<AActor> TargetActor = nullptr;
    
    UPROPERTY()
    float ElapsedTime = 0.0f;
};

// Task 本体
USTRUCT(meta = (DisplayName = "My Custom Task"))
struct FMyCustomTask : public FStateTreeTaskCommonBase
{
    GENERATED_BODY()
    
    using FInstanceDataType = FMyTaskInstanceData;
    
    virtual const UStruct* GetInstanceDataType() const override 
    { 
        return FInstanceDataType::StaticStruct(); 
    }
    
    virtual EStateTreeRunStatus EnterState(
        FStateTreeExecutionContext& Context, 
        const FStateTreeTransitionResult& Transition) const override
    {
        FInstanceDataType& Data = Context.GetInstanceData(*this);
        Data.ElapsedTime = 0.0f;
        return EStateTreeRunStatus::Running;
    }
    
    virtual EStateTreeRunStatus Tick(
        FStateTreeExecutionContext& Context, 
        const float DeltaTime) const override
    {
        FInstanceDataType& Data = Context.GetInstanceData(*this);
        Data.ElapsedTime += DeltaTime;
        
        if (Data.ElapsedTime >= Duration)
        {
            return EStateTreeRunStatus::Succeeded;
        }
        return EStateTreeRunStatus::Running;
    }
    
    UPROPERTY(EditAnywhere, Category = "Parameter")
    float Duration = 1.0f;
};
```

**方式二：Blueprint Task**

继承 `UStateTreeTaskBlueprintBase`，在蓝图中覆写：
- `ReceiveEnterState`
- `ReceiveExitState`
- `ReceiveTick`
- `ReceiveStateCompleted`

Blueprint Task 基于 UObject，比 Struct Task 占用更多内存，但对动画 TA 来说更友好——不需要写 C++。

enum class EStateTreeExpressionOperand : uint8

## 第九章：运行状态与生命周期

### 9.1 EStateTreeRunStatus

```cpp
enum class EStateTreeRunStatus : uint8
{
    Running,   // 仍在运行
    Stopped,   // 被请求停止（非成功/失败）
    Succeeded, // 执行成功
    Failed,    // 执行失败
    Unset,     // 未设置
};
```

### 9.2 状态变化类型

```cpp
enum class EStateTreeStateChangeType : uint8
{
    None,      // 非激活
    Changed,   // 状态被激活或停用
    Sustained, // 状态是新活跃状态的父状态，维持不变
};
```

**📐 概念原理 — Changed vs Sustained**：
当从 `Combat > Attack > Slash` 转换到 `Combat > Attack > Thrust` 时：
- `Combat`：Sustained（父状态维持）
- `Attack`：Sustained（父状态维持）
- `Slash`：Changed（退出）
- `Thrust`：Changed（进入）

这意味着 `Combat` 和 `Attack` 上的 Task 不会收到 ExitState/EnterState 调用，节省了不必要的重复初始化。

### 9.3 完整生命周期

```
1. UStateTreeComponent::BeginPlay()
   └── StartLogic()
       └── FStateTreeExecutionContext::Start()
           ├── 初始化实例数据
           ├── 启动 Global Evaluators
           ├── 启动 Global Tasks
           └── 选择初始状态
               └── EnterState (从根到叶)

2. 每帧 TickComponent()
   └── FStateTreeExecutionContext::Tick(DeltaTime)
       ├── TickUpdateTasks()
       │   ├── 绑定属性复制
       │   ├── Tick Evaluators
       │   ├── Tick Global Tasks
       │   └── Tick State Tasks
       └── TickTriggerTransitions()
           ├── 检查转换条件
           ├── ExitState (从叶到根)
           ├── 选择新状态
           └── EnterState (从根到叶)

3. UStateTreeComponent::EndPlay()
   └── StopLogic()
       └── FStateTreeExecutionContext::Stop()
           ├── ExitState (所有活跃状态)
           ├── 停止 Global Tasks
           └── 停止 Global Evaluators
```

## 第十章：事件系统

### 10.1 FStateTreeEvent

State Tree 的事件系统基于 `FGameplayTag` + 可选 Payload：

```cpp
struct FStateTreeEvent
{
    FGameplayTag Tag;           // 事件标签
    FInstancedStruct Payload;   // 可选的数据载荷
    FName Origin;               // 事件来源
};
```

### 10.2 事件驱动的转换

在 Transition 上设置 `Trigger = OnEvent`，并指定 `RequiredEvent`（包含 Tag 和可选的 PayloadStruct），当匹配的事件被发送时转换触发。

**实战示例**：

```cpp
// 从 C++ 或蓝图发送事件
StateTreeComponent->SendStateTreeEvent(
    FGameplayTag::RequestGameplayTag("Monster.TakeDamage"),
    FConstStructView::Make(FDamageEventPayload{ 50.0f, EDamageType::Physical }),
    "CombatSystem"
);

// 在 StateTree 编辑器中：
// Transition: OnEvent("Monster.TakeDamage") → HitReaction State
```

### 10.3 事件消费

`FStateTreeEventDesc` 有一个 `bConsumeEventOnSelect` 标志（默认 `true`）：
- 如果为 `true`，事件在状态选择成功后被消费，后续状态无法再次响应
- 如果为 `false`，事件保留，允许多个转换响应同一事件

## 第十一章：属性绑定（Property Binding）

### 11.1 绑定机制

State Tree 的属性绑定是**编译期**确定的，运行时通过 `FStateTreePropertyBindings` 执行高效的内存复制：

1. **编辑器**：在 StateTree 编辑器中，拖线将一个 Task 的输出属性绑定到另一个 Task 的输入属性
2. **编译**：PropertyBindingCompiler 将绑定关系编译为 `FPropertyBindingCopyInfo`（源偏移 + 目标偏移 + 大小）
3. **运行时**：每帧 Tick 前，通过 memcpy 批量复制绑定数据

### 11.2 外部数据（External Data）

Task 可以通过 `TStateTreeExternalDataHandle<T>` 声明对外部数据的依赖：

```cpp
struct FMyTask : public FStateTreeTaskBase
{
    // 声明需要 AIController
    TStateTreeExternalDataHandle<AAIController> AIControllerHandle;
    
    virtual bool Link(FStateTreeLinker& Linker) override
    {
        Linker.LinkExternalData(AIControllerHandle);
        return true;
    }
    
    virtual EStateTreeRunStatus EnterState(...) const override
    {
        // 获取外部数据
        const AAIController& AIController = Context.GetExternalData(AIControllerHandle);
        // ...
    }
};
```

## 第十二章：Linked State 与模块化

### 12.1 Linked State

`EStateTreeStateType::Linked` 允许一个状态引用树中的另一个 Subtree，实现**树内复用**。

### 12.2 Linked Asset

`EStateTreeStateType::LinkedAsset` 更强大——引用另一个 `UStateTree` 资产，实现**跨资产模块化**。

**实战场景**：
- 通用的 Hit Reaction 行为做成独立的 StateTree 资产
- 不同怪物的主 StateTree 通过 LinkedAsset 引用同一个 HitReaction StateTree
- 通过 `LinkedStateTreeOverrides` 可以在运行时替换引用

```cpp
// 运行时替换链接的 StateTree
StateTreeComponent->AddLinkedStateTreeOverrides(
    FGameplayTag::RequestGameplayTag("State.HitReaction"),
    FStateTreeReference(EliteHitReactionTree)
);
```

### 12.3 Run Parallel Tree

`FStateTreeRunParallelStateTreeTask`（DisplayName: "Run Parallel Tree"）允许在当前状态中**并行运行**另一棵 StateTree。

- 与 Linked Asset 的区别：Linked Asset 替代当前状态，Parallel Tree 与当前状态共存
- 用途：怪物在攻击的同时运行一个独立的"目标跟踪"树

### 13.1 State Tree → Anim State Machine 的协作模式

## 第十四章：Utility AI 与 Consideration

### 14.1 Consideration 系统

State Tree 内置了 **Utility AI** 支持。每个状态可以设置 `Considerations`，在状态选择时计算 Utility 分数：

- `SelectionBehavior = TrySelectChildrenWithHighestUtility`：选择分数最高的
- `SelectionBehavior = TrySelectChildrenAtRandomWeightedByUtility`：按分数加权随机

### 14.2 内置 Consideration 类型

包括常量分数、距离评估、曲线映射等，通过组合 `And`（取 min）、`Or`（取 max）、`Multiply`（相乘）来构建评分公式。

## 第十六章：性能优化

### 16.1 Scheduled Tick（定时 Tick）

State Tree 支持**非每帧 Tick**：
- Schema 设置 `IsScheduledTickAllowed() = true`
- 状态设置 `CustomTickRate`（如 0.1 = 每 100ms Tick 一次）
- 多个活跃状态有不同 CustomTickRate 时，取最小值

**实战建议**：远处的怪物可以降低 State Tree 的 Tick 频率，近处的保持每帧。

### 16.2 Event-Only Tick

设置 Task 的 `bShouldCallTick = false` + `bShouldCallTickOnlyOnEvents = true`，Task 仅在收到事件时被 Tick，大幅减少无效计算。

### 16.3 Struct vs UObject

State Tree 的 Task/Condition/Evaluator 都是 `USTRUCT`（不是 `UCLASS`），带来两个性能优势：
1. 更紧凑的内存布局（没有 UObject 的开销）
2. 更少的 GC 压力

## 第十八章：怪物 State Tree 实战架构

### 18.1 推荐的怪物 State Tree 结构

```
Root (Group, TrySelectChildrenInOrder)
├── Dead (State, TryEnterState)
│   └── EnterCondition: Health <= 0
│   └── Task: PlayDeathMontage → Succeeded
│   └── Transition: OnStateSucceeded → Tree Succeeded
│
├── HitReaction (LinkedAsset → HitReaction.uasset)
│   └── EnterCondition: OnEvent("Monster.TakeDamage")
│
├── Combat (Group, TrySelectChildrenWithHighestUtility)
│   ├── Attack_Melee (State)
│   │   └── Consideration: Distance < 200 → Score 0.8
│   │   └── Task: PlayAttackMontage
│   │   └── Transition: OnStateCompleted → Parent (Combat 重新选择)
│   │
│   ├── Attack_Range (State)
│   │   └── Consideration: Distance 200~800 → Score 0.7
│   │   └── Task: PlayRangeAttackMontage
│   │
│   └── Chase (State)
│       └── Consideration: Distance > 200 → Score 0.5
│       └── Task: MoveTo (TargetActor)
│       └── Transition: OnTick + Distance < 200 → Attack_Melee
│
└── Idle (State, TryEnterState)
    └── Task: PlayIdleAnimation
    └── Transition: OnTick + HasTarget → Combat
```

### 18.2 事件驱动 vs Tick 驱动

| 方式 | 适用场景 | 性能 |
|------|---------|------|
| OnEvent 转换 | 受击、死亡、技能释放等**离散事件** | ✅ 高效，不每帧检查 |
| OnTick 转换 | 距离检测、血量阈值等**连续条件** | ⚠️ 每帧检查，配合 Scheduled Tick 使用 |
| OnStateCompleted 转换 | 攻击结束回到待机等**状态完成后**的流转 | ✅ 高效 |

## 第十九章：注意事项与常见陷阱

### 19.1 State Tree 没有内置 Mover 集成

**🔧 引擎实现确认**：搜索引擎源码，State Tree 中没有任何与 Mover 插件集成的 Task。`StateTreeMoveToTask` 使用的是传统 `UAITask_MoveTo`（基于 Navigation 系统），不是 Mover 组件。

如果你想在 State Tree 中使用 Mover，需要自定义 Task。

### 19.2 Task 的 const 约束

注意 Task 的所有虚函数（EnterState、ExitState、Tick 等）都是 `const` 的——Task 本身是只读的，所有可变状态都存在 InstanceData 中。这是设计上的约束，保证 Task 可以安全地被多个实例共享。

### 19.3 Linked Asset 的 Schema 兼容性

Linked Asset 引用的外部 StateTree 必须使用**相同或兼容的 Schema**。如果 Schema 不匹配，链接会失败。
